# トレース・フェイルモデルによる要件〜検証フレームワーク

## 概要

ソフトウェア開発の失敗の多くは「要求・仕様・設計・テストが同じ対象を別々の言語で語っているため、整合性の確認が困難になる」ことに起因する。この問題は、4つの成果物すべてを「許容されるイベント列（トレース集合）と許容される停止状態（フェイル集合）」という単一の数学的枠組みで定義することで解決できる。Jackson・Hoare(CSP)・Meyer(Design by Contract) の理論を統合したフレームワークであり、この共通言語によって「設計が仕様を満たすか」「テストが仕様の検証として十分か」が形式的に判定可能になる。

## 詳細

### 基本概念

| 用語 | 定義 |
|------|------|
| **イベント** | 状態遷移の引き金。複数の状態機械が同時遷移する場合を「同期イベント」と呼ぶ |
| **トレース** | 初期状態から連続遷移可能なイベント列（内部イベントτを除く） |
| **安定状態** | τで遷移できない状態 |
| **フェイル（失敗）** | 安定状態に到達するトレースと、その状態で許容されるイベント集合の組 |

### 4成果物の定義

| 成果物 | 定義 |
|--------|------|
| **機能要求** | 「禁止トレース（悪いトレース）」と「禁止フェイル（悪いフェイル）」で記述する。例：「余った金銭を返却しないこと」は禁止フェイルとして記述 |
| **非機能要求** | 定量的指標でのみ記述できる性質（応答時間・可用率など）。「使いやすい」のような定性的記述は要件として機能しない |
| **仕様** | トレース集合とフェイル集合の組。実装方法は規定せず複数の設計を許容する自由度を残す |
| **設計** | 仕様をコンポーネント仕様に分解し、並行合成でトップレベル仕様を満たすよう構成する。各状態にガード条件・事後条件を付与 |

### 設計が仕様を満たす2条件

共通言語であるトレース集合・フェイル集合を使うからこそ、「設計が仕様を満たすか」という条件が集合の包含関係として形式的に表現できる。設計の並行合成が仕様を満たすためには以下が必要:

- **設計トレース集合 ⊇ 仕様トレース集合**
- **設計フェイル集合 ⊇ 仕様フェイル集合**

この推移性により「コンポーネント設計が仕様を満たせば、その実装の並行合成も仕様を満たす」ことが保証される。

### 各段階で自由度が保存される

仕様の段階で実装の自由度が残り、設計の段階でも実装の自由度は完全には消えない。各段階は可能性の絞り込みであって、決定ではない。

### 検証はサンプリングである

要求を「禁止トレース・禁止フェイル」として定義したことで、テストは「その禁止集合からサンプルを選んで実際に起きないことを確認する行為」と定義できる。これは核心の「共通枠組みが整合性確認を可能にする」という原理が検証フェーズにも適用された結果であり、テストケースの選択基準が明確になる。テストは「トレース集合とフェイル集合の中からサンプルを選ぶ行為」と定義される。完全な枚挙は原理的に不可能であるため、検証の本質は**リスクを最小化するサンプル選択**にある。

テストケースの種類:
- **計画主導テスト**: 設計から体系的にサンプリング
- **探索的テスト**: 経験・直感に基づく追加サンプリング
