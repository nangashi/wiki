# LLM Wiki パターン

## 概要

知識ベース維持のコストを決めるのは「読む作業」ではなく「相互参照の台帳管理」であり、LLMはその台帳作業を人間より安価に実行できる——この非対称性を活かしてwikiの更新・整合性維持をLLMに委譲し、人間はソース選択と問いかけに専念する設計パターン。Andrej Karpathyが提案。

## 詳細

### 三層アーキテクチャ

| 層 | 役割 |
|----|------|
| **Raw Sources** | 変更されない元データ（記事・論文・画像） |
| **Wiki** | LLMが生成・更新するMarkdownファイル群（エンティティ・概念・合成） |
| **Schema** | wikiの構造とワークフローを定義する設定ドキュメント（CLAUDE.md / AGENTS.md） |

### 3つの基本操作

**Ingest**（取り込み）: 新しいソースを1件ずつ処理し、LLMが内容を読んで10〜15ページを作成・更新する。

**Query**（問い合わせ）: 関連ページを検索して回答を合成し、新たな知見はwikiへ書き戻す。

**Lint**（健全性チェック）: 定期的に矛盾・古い記述・孤立ページ・情報の欠落を検出する。

### 補助インフラ

- **index.md** — カテゴリ別に整理された1行サマリのカタログ
- **log.md** — タイムスタンプ付きの追記専用ログ
- **オプションツール** — BM25/ベクトルハイブリッド検索（qmd）、Obsidianプラグイン、Marpによるスライド生成

### 設計上の洞察

「知識ベース維持の面倒な部分は読むことではなく、台帳管理だ。」LLMは相互参照の更新・整合性維持・矛盾検出といった機械的作業に長け、人間は選別と分析に専念できる。

1. **Persistence over retrieval** — 台帳管理をLLMに委譲するから知識を毎回再導出せずに済み、蓄積が複利的に成長する
2. **Schema as specification** — 設定ファイルがLLMの振る舞いを規定し、台帳管理の品質を一定に保つ
3. **Human-LLM 分業** — 人間が考え、LLMが台帳作業を実行するという非対称分業がこのパターンの前提
4. **モジュール性** — すべての要素はオプション。ドメインが要求する台帳作業の範囲に合わせて選択する

### ユースケース

- 個人の知識追跡（目標・健康・心理）
- 数週〜数ヶ月にわたる研究深掘り
- 書籍コンパニオンwiki（登場人物・テーマ・つながり）
- Slack/会議/ドキュメントを入力とするチームwiki
- 競合分析、旅行計画、趣味の探求

### 実装例（コミュニティ）

- **ΩmegaWiki** — 型付き知識グラフ・23のClaude Codeスキル
- **sqz** — コンテキスト圧縮（平均24.7%トークン削減）
- **expo-llm-wiki** — 階層型メモリ（Facts / Working Memory / Wisdom）
- **wikova.com** — オートヒーリングループを持つライブ実装
