# 6層の生産性レイヤーモデル

## 概要

個人・チーム・組織・事業の生産性を6つの層に分解し、各層間には自動的な変換が起きない「抵抗」が存在するというフレームワーク。AIツール導入が個人層にとどまり組織層に波及しない現象を構造的に説明する。

## 詳細

### 6つの層

| 層 | 名称 | 内容 |
|----|------|------|
| L1 | ツール導入層 | Copilot・Claude等のAIツール、CI/CDなどのインフラ |
| L2 | 個人生産性層 | 個人の実装速度・タスク処理効率 |
| L3 | 接続・調整層 | **ボトルネック地帯**。レビュー待ち・仕様調整・承認フロー |
| L4 | チーム・プロセス層 | スプリント運営・WIP制限・知識共有の仕組み |
| L5 | 組織デリバリー層 | リリース頻度・変更失敗率・復旧時間（DORA Metrics） |
| L6 | 事業価値層 | 顧客への価値提供・売上・競争力 |

**核心原則**: 下位層の改善は上位層に**自動的には伝播しない**。L1を最適化してもL3の待ち行列が詰まっていれば組織速度は変わらない。

### なぜこの6層で切るか（分類軸の根拠）

層の境界は恣意的な粒度分けではなく、「改善の主体」と「制約の所在」が入れ替わる位置に置かれている。L1→L2は道具から個人の使いこなしへ、L2→L3は個人作業から他者との調整へ、L3→L4は個別調整からプロセスの仕組みへ、L4→L5はチーム運営から組織全体の統合へ、L5→L6は組織内部の効率から市場という外部の評価者へと、それぞれ主体と制約条件が切り替わる。改善が自動変換されない「抵抗」はまさにこの切り替わりの位置に発生するため、この軸で切ると「改善がどの層で失速したか」を特定できる。隣接層を同一視する（例: 個人が速くなった＝チームが速くなった）と、失速箇所の診断ができなくなる。

### 観察された逆説

- 個人開発者の体感: AI導入で「速くなった」(約+20%)
- 実態データ: 組織のデリバリー安定性は負の相関(約−19%)

この乖離は、現状のAI投入が **L2（個人層）に集中している** ことで説明できる。L3の調整コストが解消されない限り、L1〜L2の改善は滞留する。

### ツールはミラーでありマルチプライヤー

この逆説を別の角度から言い換えると、ツールは「現状を映す鏡（ミラー）」であり「現状を増幅する乗数（マルチプライヤー）」として機能する。

```
組織の成果 = 組織の基礎体力 × ツールの効果
```

基礎体力が厚ければツールが強みを加速し、基礎体力が薄ければ問題が拡大される。L3〜L5が機能していない組織でAIを導入すると、ミラーとして「調整コストの高さ」を可視化し、マルチプライヤーとして「個人が速くなるほど調整の摩擦が増す」という逆説を拡大する。

### AI投入の現状分布

収集した40超の事例をライフサイクル工程（戦略〜運用）×スコープ（個人/チーム/組織）でマトリクス化すると、投入の大部分が **「実装×個人」** 象限に集中している。

### 組織の基礎体力（4要素）

L3〜L5を機能させるには以下の基盤が必要：

1. **自動化** — CI/CD、テスト自動化、IaC
2. **プロセス設計** — アジャイル、WIP制限
3. **組織設計** — チーム構造、知識共有の仕組み
4. **計測** — DORA Metrics、バリューストリームマッピング

### 診断・介入の手順

核心原則は「下位層の改善は上位層に自動的には伝播しない」である。この介入手順は、核心の「抵抗」が存在する層を特定し、その層の抵抗を直接解消することで改善を上位層に伝播させることを意図している。

1. **現状診断**: 3スケール層で詰まり箇所を可視化 — 抵抗がどの層に存在するかを特定する。これを省くと、改善がL3の抵抗に阻まれたまま伝播しない
2. **問題構造化**: バリューストリームで作業と待機を判別 — L3（接続・調整層）の待ち行列という抵抗の正体を具体化する
3. **介入点選択**: ツール追加（浅）→ 組織設計（中）→ 前提転換（深） — 抵抗の強さに応じた干渉の深さを選ぶ。ツール追加はL1〜L2の改善、組織設計はL3〜L4の抵抗解消、前提転換はL5〜L6の構造変更に対応する
4. **レバレッジ設計**: 複数面の掛け合わせで制約を順次解消 — 抵抗は単層ではなく複数層に存在することが多いため、上位層への伝播経路を複数確保する
