# Wiki スキーマ

このファイルはwikiの全ルールと共通手順を定義する。`/ingest`, `/query`, `/lint` スキルは必ずこのファイルを最初に読む。

---

## ディレクトリ構成

```
wiki/
  collections.md     # コレクション定義（全スキル共通）
  insight/
    schema.md        # このファイル（ルール定義）
    pages/           # wikiページ（1概念1ページ）
      <slug>.md
    references/      # 審査基準・共通参照ドキュメント（スキルから参照）
      <name>.md
  it/
    schema.md        # ITコレクションのルール定義
    pages/           # ITコレクションのページ
```

## ページフォーマット

### frontmatter

```yaml
---
title: "概念名（日本語）"
sources:
  - "https://example.com/page"
  - "book:書名（著者名）"
  - "topic:トピック名"
created: "YYYY-MM-DD"
updated: "YYYY-MM-DD"
---
```

### 本文構成

```markdown
# 概念名

## 概要
1〜3文で概念の核心を説明する。

## 詳細
詳しい説明、背景、仕組み、使い方など。

## 関連
- [[関連概念スラグ]] — この概念との関係を1行で説明
- [[別の概念スラグ]] — この概念との関係を1行で説明
```

- セクションは必要なものだけ使う（例：「詳細」不要なら省略）
- 追加セクション（例：「具体例」「注意点」「歴史」）は内容に応じて自由に追加してよい

### 本文中のリンク記法

本文（詳細セクションなど）の流し文章中でリンクを張る場合は、`概念名（[[slug]]）` の形式を使う。

```markdown
# NG: エイリアス記法（[[slug|表示テキスト]]）は使わない
生産性の[[proxy-metrics-knowledge-work|代理指標]]になる。

# OK: 日本語を先に書き、括弧内にリンクを置く
生産性の代理指標（[[proxy-metrics-knowledge-work]]）になる。
```

- 概念名がすでに括弧付き表記を含む場合（例: `制約理論（TOC）`）は `制約理論（TOC・[[theory-of-constraints]]）` のように区切り文字を使う
- 関連セクションでは `[[slug]]` のみで十分（タイトルは説明文で補う）

---

## 命名規則

- **ファイル名**: 英語またはローマ字表記の kebab-case（例: `transformer-architecture.md`, `attention-kikou.md`）
- **タイトル**: 日本語（frontmatter の `title` フィールドおよび本文 `# 見出し`）
- **スラグ**: ファイル名から `.md` を除いたもの（例: `transformer-architecture`）
- **タグ**: 日本語OK、分野・種別・用途で分類

## 粒度ルール

### 1概念1ページの原則
- **1ページ = 単一のアイデア・技術・用語・手法・人物、または複数概念の間の関係・対比・パターン（関係概念）**
- ページを読んで「これは複数のことを説明している」と感じたら分割候補
- ページが2〜3行しかなく独立性が低ければ、関連ページへの追記候補

**関係概念の例（通常の概念ページと同じフォーマットで作成する）:**
- XとYの定義アプローチの対比
- 複数研究にまたがる評価パラダイムの分類
- 同一問題に対する複数手法のトレードオフ比較
- ※ 関係概念ページは `sources` に複数のソースが並ぶことが多い

### 新規 vs 追記の判断フロー
```
抽出した情報 → 同名または非常に類似した既存ページがある？
  YES → そのページに情報が不足している？
    YES → 既存ページに追記
    NO  → 既に記録済み、スキップ
  NO  → 独立した概念として成立する？
    YES → 新規ページを作成
    NO  → 最も関連するページに追記
```

### 既存ページとの類似性チェック
- 完全同一概念: 同じページに追記（重複ページを作らない）
- 上位概念: 上位ページの「関連」に子概念へのリンクを追加し、子概念ページを新規作成
- 類似だが別概念: それぞれ別ページとして作成し、相互リンク

## ソース記録ルール

frontmatter の `sources` に記録する形式:

| 取込元 | sources の記述形式 |
|--------|------------------|
| 特定のWebページ | `"https://..."` |
| 特定のPDF | `"file:ファイル名.pdf"` |
| 本に関するWeb収集 | `"book:書名（著者名）"` + 収集したURL群 |
| トピックに関するWeb収集 | `"topic:トピック名"` + 収集したURL群 |

Web収集の場合、特殊マーカー（`book:` / `topic:`）を **先頭** に置き、その後に実際に参照したURLを列挙する。

## 共通書き込み手順

この手順を `/ingest`・`/query` の両スキルで使用する。

### ステップ1: 既存ページを把握する
`wiki/index.md` を読み、すでに記録されている概念とその内容を把握する。

index.md が空の場合は以下で一覧を確認する:
```
find wiki/insight/pages -name "*.md" | sort
```

### ステップ1.5: 関係概念を探す
index.md のサマリをもとに、今回のソースと既存ページの間に以下が成立しないか検討する:

1. **対比**: 同じ問いに対して異なるアプローチをとっているページと今回のソースの組み合わせ
2. **パターン**: 複数のページ・ソースに共通する傾向や分類軸
3. **統合枠組み**: 複数の既存ページをまとめて整理できる上位概念

該当する場合、関係概念ページを作成候補としてステップ2に追加する。  
関係概念はステップ3以降の判断フローを通常の概念と同じように処理する。

### ステップ1.6: 合成元ページの価値評価（関係概念ページを作成した場合のみ）

ステップ1.5で関係概念ページを新規作成した場合、合成元として参照したwikiページそれぞれを評価する。

1. **独自性の確認**: 合成ページに取り込まれていない独自の内容が残っているか
2. **参照状況の確認**: 他ページからリンクされているか

| 独自内容 | 参照状況 | 対応 |
|---------|---------|------|
| あり | あり | **保持** — 合成ページへのリンクを追加するだけ |
| あり（少量） | なし/少ない | **追記に落とす候補** — 独自内容を合成ページに追記し元ページを削除 |
| なし | なし/少ない | **削除候補** — 元ページを削除 |

「追記に落とす」または「削除」を提案する場合はユーザーに確認を取ってから実行する。

確認メッセージ例:
```
[[元ページスラグ]] の内容は [[合成ページスラグ]] に包含されています。
削除してよいですか？（独自内容: なし / 参照元: なし）
```

### ステップ2: 取り込む概念を列挙する
ソースから抽出した情報を「概念単位」に分解する。  
**目安**: 1つの概念 = 独立して説明できる1つのアイデア

**ソースが言語・ツール固有の場合**: ソースが特定言語（Go・Python等）・フレームワーク・ツールに特化した記事（技術ブログ・実装チュートリアル等）の場合、各概念について「この概念は言語・ツールを外しても成立するか」を確認する。成立する場合は概念を言語非依存の形で再定義してから列挙する。言語固有の実装詳細（API名・コマンド・構文）は概念そのものではなく、後工程で「具体例」として扱う。

### ステップ2.5: 各概念のコンテンツ型を分類する

各概念に以下のいずれかの型を割り当てる。型によって核心の抽出方法とステップ4の書き方が変わる。

| 型 | 判断の目安 | 核心を見つける問い |
|---|---|---|
| **原理・原則型** | 「なぜXが起きるか」を説明している | そのままWhy/Howを問う |
| **分類・タクソノミー型** | 事象をカテゴリに整理している | なぜこの分類軸が有効か。分類が前提とする原理は何か |
| **パターン・実践型** | 設計パターン・ベストプラクティスを紹介している | このパターンはどんな問題を解くか。なぜこのアプローチが有効か |
| **理論・フレームワーク型** | 他者の理論・フレームワークを要約している | 著者の問題意識・提唱動機は何か（これが核心。不明なら補足調査） |
| **ハウツー・実装型** | 手順・実装方法を説明している | 手順の背後に非自明な原理があるか（→採用判断の分岐あり） |
| **観察・事例型** | 特定の経験・実験から得た知見である | 一般化可能な原理まで引き上げられるか（→補足調査の分岐あり） |

#### 型ごとの注意点

**分類・タクソノミー型**  
分類と同時に「なぜこの分類が有効か」を必ず抽出する。原理がなければ参照表にとどまる。概要には分類の定義より先に「この分類を成立させる原理」を書く。

**パターン・実践型**  
「何ができるか（what）」より「なぜ有効か（why）」を優先して書く。パターンを生み出した「解くべき問題」が核心になることが多い。

**理論・フレームワーク型**  
著者の提唱動機がソースに明示されていない場合は補足検索を行い、動機を特定してから書く。動機が不明のまま書いたページは本質が推測補完になるリスクがある。

**ハウツー・実装型（採用判断の分岐）**  
```
手順の背後に非自明な原理がある？
  YES → 原理を核心として書く。手順は原理を体現する具体例として位置づける
  NO  → 採用しない候補として扱う（下記の提案フォーマットでユーザーに確認）
```

提案フォーマット:
```
「<概念名>」はハウツー型のコンテンツですが、一般化可能な原理が見当たりません。
採用しない場合は技術メモとしてコード・設定コメントに残すことを推奨します。
採用しますか？
```

**観察・事例型**  
特定の経験から一般化可能な原理（「この状況では常にXが起きる、なぜなら…」）まで引き上げられるか確認する。引き上げられない場合は補足調査を提案する。一般化できない観察は事例ページとしてではなく、関連する原理ページへの追記として吸収することを検討する。

### ステップ3: 各概念について新規 or 追記を判断する
上記「粒度ルール > 新規 vs 追記の判断フロー」に従う。

### ステップ4: ページを書く・更新する

**新規ページの場合:**
- ファイル名: 命名規則に従い `wiki/insight/pages/<slug>.md` として作成
- frontmatter: title・sources・created・updated を記入
- 本文: 概要 → 詳細 → 関連 の順で記述
- **概要**: ステップ2.5 で特定した型の「核心を見つける問い」に対する答えを1文で書いてから、説明・背景に続ける（分類や手順の定義を先置きしない）
- **詳細**: 概念の普遍的な構造・原理・メカニズムを書く場所。特定言語・ツール固有の実装（コード・コマンド・API名）はデフォルトで「具体例」セクションへ分離し、最小限の1例にとどめる。詳細セクションを読んで「これはGoの話だ」「これはPostgreSQLだけの話だ」と感じたら、言語・ツールを外して書き直す。

**追記の場合:**
- 既存ページの `updated` を今日の日付に更新
- 新しい情報を適切なセクションに追記
- `sources` に新しいソースを追加（重複しない場合）

**index.md の更新:**
新規ページを作成した場合、または既存ページの概要が変わった場合は `wiki/insight/index.md` を更新する。
```
- [[slug]] — 1〜2文のサマリ
```

### ステップ4.5: 新規作成ページの品質評価

新規作成したページに対してのみ実施する（追記の場合はスキップ）。

以下の3観点で評価し、**2つ以上が該当する場合**に補足調査を提案する。

| 観点 | 低品質の兆候 |
|------|------------|
| **具体性** | 具体例・数値・コード・固有名（ツール名・論文名など）がなく抽象的な説明のみ |
| **情報密度** | 詳細セクションが概要の言い換えになっており、読み進めても新情報がない |
| **自立性** | このページだけで1つの問いに答えられず、ソースを直接参照しないと理解できない |

**提案の形式:**

```
### 品質評価: [[スラグ]] — タイトル

以下の点で情報が薄い状態です:
- （該当する観点とその根拠）

補足調査を行いますか？（キーワード案: 「...」「...」）
```

- ユーザーが承認した場合: その場で `WebSearch` を実行し、ページを更新してから次のステップへ進む
- ユーザーが断った場合: そのまま保持し、ステップ6のサマリに「補足調査未実施」として記録する
- 品質が十分な場合: このステップは何も出力しない

### ステップ4.6: 本質審査（新規作成ページのみ）

新規作成したページごとに **Agent ツールでサブエージェントを起動し** 本質審査と改善を行う。複数ページがある場合は並列で起動してよい。

**サブエージェントへの指示（`<slug>` を実際のスラグに置き換えて渡す）:**

```
本質審査を実施する。

1. `wiki/references/essence-review-criteria.md` を読んで審査基準を確認する
2. `wiki/pages/<slug>.md` を読む
3. C1・C2・C3 の判定を行う
4. 改善が必要な場合（C1 または C2 が不合格、あるいは対処セクションあり・C3 不合格）:
   - Edit ツールでページを直接改善する
   - 改善した基準・箇所・追加内容を報告する
5. 全基準合格の場合は「合格」とのみ報告する
```

**結果の扱い:**
- サブエージェントが改善した場合、改善内容をステップ6のサマリに記録する
- 合格の場合は「合格」とのみ記録する

### ステップ5: 関連リンクを相互に張る
- 新規・更新ページの「関連」セクションに関連概念へのリンクを追加
- 関連先ページの「関連」セクションにも逆リンクを追加
- リンク形式: `[[スラグ]]` + 関係の説明

### ステップ6: 作業サマリを出力する
以下を報告する:
- 新規作成したページ一覧（タイトル・スラグ）
- 更新したページ一覧（タイトル・追加内容の要約）
- スキップした概念（理由付き）
- 追加した関連リンク
- 本質審査の結果（合格 / 改善内容の要約）
- 補足調査未実施のページ（ユーザーが断った場合のみ）
