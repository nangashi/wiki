# LLM知識システムパターン（Karpathy）

## 概要

知識ベース維持のコストを決めるのは「読む作業」ではなく「相互参照の台帳管理」であり、LLMはその台帳作業を人間より安価に実行できる——この非対称性を活かしてwikiの更新・整合性維持をLLMに委譲し、人間はソース選択と問いかけに専念する設計パターン。Andrej Karpathyが提案。

## 仕組み

### 三層アーキテクチャ

| 層 | 役割 |
|----|------|
| **Raw Sources** | 変更されない元データ（記事・論文・画像） |
| **Wiki** | LLMが生成・更新するMarkdownファイル群（エンティティ・概念・合成） |
| **Schema** | wikiの構造とワークフローを定義する設定ドキュメント（CLAUDE.md / AGENTS.md） |

### 3つの基本操作

**Ingest**（取り込み）: 新しいソースを1件ずつ処理し、LLMが内容を読んで10〜15ページを作成・更新する。

**Query**（問い合わせ）: 関連ページを検索して回答を合成し、新たな知見はwikiへ書き戻す。

**Lint**（健全性チェック）: 定期的に矛盾・古い記述・孤立ページ・情報の欠落を検出する。

### 補助インフラ

- **index.md** — カテゴリ別に整理された1行サマリのカタログ
- **log.md** — タイムスタンプ付きの追記専用ログ
- **オプションツール** — BM25/ベクトルハイブリッド検索（qmd）、Obsidianプラグイン、Marpによるスライド生成

### 設計上の洞察

「知識ベース維持の面倒な部分は読むことではなく、台帳管理だ。」LLMは相互参照の更新・整合性維持・矛盾検出といった機械的作業に長け、人間は選別と分析に専念できる。

1. **Persistence over retrieval** — 台帳管理をLLMに委譲するから知識を毎回再導出せずに済み、蓄積が複利的に成長する
2. **Schema as specification** — 設定ファイルがLLMの振る舞いを規定し、台帳管理の品質を一定に保つ
3. **Human-LLM 分業** — 人間が考え、LLMが台帳作業を実行するという非対称分業がこのパターンの前提
4. **モジュール性** — すべての要素はオプション。ドメインが要求する台帳作業の範囲に合わせて選択する

### ユースケース

- 個人の知識追跡（目標・健康・心理）
- 数週〜数ヶ月にわたる研究深掘り
- 書籍コンパニオンwiki（登場人物・テーマ・つながり）
- Slack/会議/ドキュメントを入力とするチームwiki
- 競合分析、旅行計画、趣味の探求

### コミュニティ実装例

- **ΩmegaWiki** — 型付き知識グラフ・23のClaude Codeスキル
- **sqz** — コンテキスト圧縮（平均24.7%トークン削減）
- **expo-llm-wiki** — 階層型メモリ（Facts / Working Memory / Wisdom）
- **wikova.com** — オートヒーリングループを持つライブ実装

## 他のLLM知識管理アプローチとの比較

| | LLM維持wiki（本パターン） | RAG（ベクトル検索） | 生ノート＋全文検索 |
|---|---|---|---|
| 主な用途 | 長期蓄積する個人・チームの知識ベース | 大量ドキュメントへの都度参照 | 軽量な個人メモ |
| 知識の形態 | LLMが合成・構造化したページ（読める記事） | 元文書のチャンク（合成されない） | 書いたままの断片 |
| 相互参照の整合性 | Lint操作で維持される | 概念が重複・矛盾したまま並存する | 人間の手作業に依存 |
| 取り込みコスト | 高い（Ingestで毎回LLMが処理） | 低い（埋め込み生成のみ） | ほぼゼロ |
| 参照時の品質 | 事前に統合済みで一貫した回答が得やすい | 検索精度とチャンク境界に依存 | 検索ヒットの当たり外れが大きい |
| 知識の複利成長 | ページ間接続が増えるほど加速する | 蓄積しても構造は増えない | 蓄積するほど探しにくくなる |

RAGが「書き込み時は安く、読み出し時に毎回合成する」のに対し、本パターンは「書き込み時に高いコストを払い、読み出しを安く一貫させる」。参照頻度が高く長期に使う知識ほど本パターンが有利になる。

## 採用すべきケース

- 数ヶ月〜年単位で育てる知識ベース — 取り込み時の統合コストが参照のたびに回収される
- 知識間の矛盾・重複が問題になる領域 — Lintによる整合性維持はRAGにない固有の価値
- 人間が通読する用途を兼ねる — LLMが合成したページは記事として読める

## 採用しないべきケース

- 一度きりの調査・使い捨ての文書群 — Ingestコストが回収されない。RAGか単純検索で足りる
- 元文書の逐語的正確さが必須の用途（法務・規約参照等） — LLMの合成は要約・言い換えを伴うため、原文リンクの検索システムの方が安全
- 更新頻度が極端に高いデータ（メトリクス・チケット状態等） — wikiページの更新が追いつかず、常に陳腐化した状態になる。DBやダッシュボードの領分

## 実装でトレードオフが生じるケース

- **LLMへの書き込み委譲 vs 内容の信頼性**: 台帳作業の委譲はこのパターンの核心だが、LLMは取り込み時に事実の歪み・過剰な一般化を持ち込みうる。品質ゲート（審査基準・ソース格付け）を設けるほど信頼性は上がるが、委譲によるコスト削減効果は目減りする
- **ページ粒度の設計**: 細かく分けると相互参照の台帳が肥大化しLint コストが増える。大きくまとめると1ページの更新でLLMが読み書きするトークンが増える。参照単位（何を1回で読みたいか）に合わせて決める
