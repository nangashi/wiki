# 間接プロンプトインジェクション（IPI）

## 概要

LLMはコンテキスト内の文字列を送信元によらず等価に「指示」として処理するため、外部コンテンツ取得を伴うエージェントは「正規データの読み込み」と「攻撃者の指示実行」を構造的に区別できない。この性質を利用して外部テキストに悪意ある指示を埋め込み、エージェントに意図しない操作を実行させる手法。ユーザーが直接悪意あるプロンプトを入力する「直接注入」と区別される。

## 仕組み

### 攻撃経路

LLMエージェントはタスク達成のために外部コンテンツを取得・処理することが多い。攻撃者はその外部コンテンツ内に「エージェントへの指示」を隠し込む。エージェントは正規コンテンツとして処理するため、悪意ある指示を実行してしまう。

代表的な攻撃経路：
- 要約を依頼されたWebページに隠し指示を埋め込む
- GitHubリポジトリのREADMEやコードコメントに指示を挿入する
- メール本文やドキュメントファイルに指示を埋め込む

### 防御の多層構造

LLMが「命令」と「データ」を本質的に区別できない以上、単一の防御で完結する対策は存在せず、多層防御が唯一の現実解となる。各層は防げる攻撃と防げない攻撃が異なる。

| 防御層 | 手法 | 限界 |
|--------|------|------|
| サンドボックス隔離 | ネットワーク・権限をホワイトリスト化 | 応答に機密情報を含める形の窃取は防げない |
| 権限最小化 | `.env`・秘密鍵へのアクセスを明示的に拒否 | 過度に制限するとエージェントの有用性を損なう |
| 入力サニタイズ | HTMLコメント・ゼロ幅文字・既知パターンの除去 | 自然言語の攻撃パターンは無限に近く、フィルタ単独では対応不可 |
| プロンプト設計 | `<document>` 等のデリミタでデータ境界を明示（spotlighting） | 確率的な防御であり完全ではない |
| 出力検証＋人間承認 | 状態変更操作に承認プロセスを挿入 | コスト・遅延が増加する |

**入力の信頼度分離**: 上記の基盤となる設計原則。ユーザー指示（高信頼）と外部コンテンツ（最低信頼）を別の信頼レベルで扱い、処理前にデータへ優先度ラベルを付与する。システムプロンプトとWeb・メール由来テキストを同じ扱いにすることが、攻撃を成立させる設計上の見落としになる。

**Managed Settingsでの制御**: AIツールが処理できる外部入力の範囲を管理者が制限する。エージェントが取得できる外部コンテンツ自体を絞ることで、攻撃者が悪意ある指示を届ける経路を遮断する（[[claude-code-managed-settings]]参照）。

## 実装のアンチパターン

### 1. サンドボックス万能視

「隔離環境で動くから安全」と判断する。サンドボックスはコマンド実行の被害は抑えるが、応答テキストに機密情報を紛れ込ませて流出させる攻撃（データの窃取）は防げない。

```
Bad:  サンドボックス化したのでエージェントに社内文書とWeb取得の両方を許可する
Good: サンドボックスに加え、機密データへのアクセスと外部送信経路が同一セッションに共存しないよう境界を設計する
```

### 2. 入力フィルタリング単独依存

既知の攻撃キーワードの除去だけで対策とする。自然言語の攻撃パターンは無限に生成でき、新パターンは必ずすり抜ける（偽陰性）。フィルタリングは多層防御の一層としてのみ意味を持つ。

### 3. MCPツール定義の無検証導入

外部のMCPサーバー・ツール定義をバージョン固定やハッシュ検証なしに導入する。ツール定義自体に悪意ある指示を仕込むTool Poisoning攻撃の経路になる。

## トレードオフ

- **セキュリティ vs ユーティリティ**: 厳格な信頼境界・権限制限はエージェントの有用性を直接削る（ETH Zurich/Google/Invariant Labsの研究でも指摘）。「どこまで制限するか」は守るべき資産の価値と業務影響の比較で決めるしかない
- **検出強化 vs 遅延・コスト**: 人間承認や出力検証の挿入は安全性を上げるが、エージェントの自律性という価値そのものを部分的に打ち消す

## 事例・運用知見

- **Clinejection（GitHub Issue経由）**: GitHub ActionsでIssueタイトルをAIが処理する構成で、誰でもIssueを作成できたため、攻撃者がタイトルに埋め込んだ命令でnpmトークンが窃取され、悪意あるパッケージが約4,000台に配布された
- **McKinsey「Lilli」**: プロンプトレイヤーとデータレイヤーが分離されておらず、SQLインジェクション経由でシステムプロンプトが書き換え可能だった（利用者43,000人規模）
- **HashJack（URLフラグメント）**: URLの `#` 以降はサーバーに送信されないため、WAF・サーバーサイド対策をすり抜けて攻撃命令を運べる
- **RAGドキュメントポイズニング**: 社内WikiのHTMLコメントに命令を埋め込む。削除されるまで永続的に影響し、1つの毒入りドキュメントが全ユーザーに波及する
