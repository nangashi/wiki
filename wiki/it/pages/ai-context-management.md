# AIへのコンテキスト設計

## 概要

LLMはコンテキスト内の情報を真として処理する確率的変換器であり、渡す内容の質・構造・完全性がそのまま出力品質に反映される。設計の焦点は「何を渡すか」に尽きる。プロンプト技術で補おうとするほど指示間の矛盾が増え、LLMの恣意的補完を招く。

## 仕組み

### LLMの構造的特性と対処

9ヶ月以上のコーディングエージェント実運用から観察された4つの限界パターン。プロンプト改善では根本解決できない。

| 特性 | 顕在化パターン | 対処 |
|------|------------|------|
| **非決定論的振る舞い** | 同一プロンプトで毎回異なるスタイルが生成され、コードベースの一貫性が崩れる | linterで書いてよいコードの空間を制約し、選択肢を削減する |
| **コンテキスト絶対化** | 誤った前提・バグのある既存コードをコンテキストとして与えると引きずられた実装を行う | 「無垢なジュニアエンジニア」として全前提を疑う問いを投げ、外部から解体する |
| **恣意的解釈** | 指示の曖昧さや矛盾を説明なく補完して実装を進める | 見積もり段階で矛盾点・疑問点の完全列挙を強制し、着手前に曖昧さを潰す |
| **知識-活用乖離** | SOLID原則・設計パターンを知っているが適用しない | eslint-plugin-boundariesで原則違反をツールレベルで検出し、「適用しない」を選択肢から外す |

### 指示の4層構造

人への指示もAIへの指示も共通の基本型。いずれかの層が欠けると推測・補完が起き、期待外れの結果になる。

| 層 | 内容 | 欠落時の症状 |
|---|---|---|
| **目的** | なぜその業務を行うか | 文脈なき作業で判断軸がなくなる |
| **業務内容** | 何をするか | スコープが曖昧で勝手に拡張・縮小される |
| **業務要件** | どのように・どの水準で | 品質基準がなく出力がバラつく |
| **参考情報** | 判断や実行に必要な背景情報 | 既存コードの文脈を無視した実装になる |

### 観点の先行条件（分析・相談タスク）

分析・相談タスクでは、観点（思考の枠組み）を先に与えないと解空間を任意に切り取った高抽象度の回答になる。

**観点なし（悪い例）**:

```
「うちの組織の生産性問題を分析して」
→ ぼんやりとした要因リスト（次のアクションが見出せない）
```

**観点あり（良い例）**:

```
「マネジメント業務の全体像（地図）を使って、生産性問題の要因を分析して」
→ 実務の心当たりに出会える粒度の具体的な仮説
```

観点の枠組み自体の設計（何が本質的かの選球眼）は現場経験を持つ人間の役割。AIが生成できる枠組みは訓練データの頻度に規定された任意の切り取りになる。

### 着手前チェックリスト（コーディングタスク）

LLMに実装を依頼する前の見積もりフェーズで確認する:

- [ ] 編集対象ファイル名と行番号を明示したか
- [ ] 関連DB情報（テーブル・カラム）を明示したか
- [ ] I/O全種類を明示したか
- [ ] 矛盾点・疑問点を完全に列挙したか（恣意的補完の余地を潰す）
- [ ] フェーズ分割を最適化したか

## 実装のアンチパターン

### 1. 観点なし相談

「何でもいいから分析して」は解空間が無限になりLLMが訓練データの頻度分布に引きずられる高抽象度の回答になる。

```
Bad:  「うちの組織の生産性問題を分析して」
Good: 「マネジメント業務の全体像（地図）を使って、生産性問題の要因を分析して」
```

### 2. 目的層の欠落

業務内容だけ渡して目的を省略すると、判断が必要な場面でLLMが推測で補完する。

```
Bad:  「このCSVをパースする関数を書いて」
Good: 「経理の月次締めで銀行明細を取り込むため、このCSVをパースする関数を書いて。
      不正行はスキップでなくエラーにする（見逃すと決算数値が狂うため）」
```

### 3. 矛盾した指示の追加

後から矛盾する指示を加えると、コンテキスト絶対化によりLLMが「両方正しい」として恣意的に解釈する。指示を追加する際は、既存指示との整合を確認し、上書きなら明示的に「先の指示Xは取り消す」と書く。
