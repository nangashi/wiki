# Wide Events（ワイドイベント）

## 概要

Wide Eventsは可観測性・デバッグのためのログ設計パターンで、リクエスト終了時に文脈を凝縮した1つの包括的イベントを発行する（Canonical Log Linesとも呼ばれる）。従来のログは記録単位がコード実行行単位のため、1リクエストのデバッグに多数の行を手作業で突き合わせる必要があるが、Wide Eventsは観察単位（リクエスト）と記録単位を一致させることで、「プレミアムユーザーの決済エラーをエラーコード別に集計」のような問いを構造クエリで即答できるようにする。

## 他の可観測性アプローチとの比較

| | Wide Events | 従来の構造化ログ | メトリクス+トレーシング |
|---|---|---|---|
| 主な用途 | リクエスト単位のデバッグ・分析 | イベント記録・監査 | パフォーマンス監視 |
| クエリの柔軟性 | ○（任意フィールドで集計） | △（フォーマット不統一になりやすい） | △（事前定義の次元のみ） |
| 高カーディナリティ対応 | ○（列指向DBで対応） | ✗（インデックスがスケールしない） | ✗（組み合わせ爆発） |
| ビジネス文脈の記録 | ○（任意フィールドを追加） | △（手動でフォーマット統一が必要） | ✗（メトリクスに文脈は乗せにくい） |
| 導入コスト | 高（列指向DB・設計変更が必要） | 低 | 中（OTel SDK導入） |

## 採用すべきケース

- **マイクロサービス・分散システム**: 1リクエストが複数サービスを横断し、従来ログでは原因特定に時間がかかる
- **高カーディナリティなデバッグ**: user_id・request_idでのフィルタリングや集計が必要
- **ビジネス文脈とエラーの横断分析**: サブスクリプション種別・決済情報などを絡めた問いに即答したい

## 採用しないべきケース

- **低トラフィックのモノリス**: 従来の構造化ログで十分カバーできる
- **ストレージコストが極めて厳しい環境**: 1イベント50フィールド以上になるため、テイルサンプリングなしでは保存コストが大きい
- **既存のメトリクス・トレーシングで問いに答えられている**: すでに可観測性が機能しているなら移行コストを正当化しにくい

## 実装のアンチパターン

### 1. 行指向DBへの保存

Wide Eventsはフィールド数が50以上になり高カーディナリティなデータを扱う。行指向DBではクエリ時に全フィールドをスキャンするため、コストとレイテンシが急増する。

```sql
-- Bad: MySQLやPostgreSQLにWide Eventsを格納
-- クエリ時に全列をスキャン → 大量データで破綻
INSERT INTO events (timestamp, request_id, ..., field_50) VALUES (...)

-- Good: 列指向DB（ClickHouse・BigQuery）を使う
-- クエリ対象の列だけを読み込むため、フィールド数が増えてもスキャンコストが増大しにくい
```

### 2. 全イベントの無差別保存

テイルサンプリングなしで全イベントを保存するとストレージコストが爆発する。

```python
# Bad: 全リクエストを保存
store_event(event)

# Good: 優先度ベースのテイルサンプリング
if event.status_code >= 500:    # エラー全件
    store_event(event)
elif event.duration_ms >= p99:  # 遅いリクエスト全件
    store_event(event)
elif is_vip_user(event):        # 特定ユーザー全件
    store_event(event)
elif random() < 0.05:           # 正常・高速リクエストの5%
    store_event(event)
```

### 3. ビジネス文脈フィールドの省略

OpenTelemetryを導入しただけで可観測性が完成したと誤解し、ビジネス文脈フィールドを追加しないケース。OpenTelemetryは「配信の仕組み」であり、何を記録するかは開発者の責任として残る。

```json
// Bad: インフラ文脈のみ
{
  "timestamp": "...",
  "service": "checkout-service",
  "status_code": 500,
  "duration_ms": 1247
}

// Good: ビジネス文脈を含む
{
  "timestamp": "...",
  "service": "checkout-service",
  "status_code": 500,
  "duration_ms": 1247,
  "user": { "id": "...", "subscription": "premium", "lifetime_value_cents": 284700 },
  "error": { "type": "PaymentError", "code": "card_declined" }
}
```

## 実装でトレードオフが生じるケース

### ストレージ選定

| アプローチ | メリット | デメリット |
|-----------|---------|-----------|
| ClickHouse（セルフホスト） | クエリ高速・コスト低 | 運用負荷あり |
| ClickHouse Cloud | マネージド・運用負荷小 | セルフホストよりコスト高 |
| BigQuery | フルマネージド・GCPと統合容易 | クエリコストが従量課金で読みにくい |
| Elasticsearch | 既存構成と統合しやすい | 高カーディナリティに弱く本来向いていない |

**推奨**: 新規導入ならClickHouse Cloud。GCP基盤がすでにある場合はBigQuery。

### テイルサンプリングのタイミング問題

複数サービスにまたがるリクエストでは、テイルサンプリングの判断タイミングにずれが生じることがある。

対策の選択肢:
- **収集バッファ**: 一定時間（例: 30秒）イベントをバッファし、リクエスト完了を確認してからサンプリング判定。精度は高いが実装が複雑
- **サービス単独判定**: 各サービスが独立してサンプリングを判断。シンプルだが一貫性が下がる
