# 中間記法パターン（MNP）

## 概要
既存ツールへのAI統合を「AIにGUIを操作させる」ではなく「GUIと同等の状態を持つ内部用DSLを設計し、AIにそのDSLを操作させる」方式に切り替えることで、実装速度・出力安定性・コストを同時に改善するパターン。

## 仕組み
従来のAI統合では、JSON/API経由でGUIと同等の操作をAIに実行させるアプローチが多く用いられるが、実装コストが数週間規模になりやすく出力の安定性にも課題がある。

MNPの核心は「AIが出力したテキストをツール側で静的に変換する」という分業設計にある。AIはGUIの内部状態を模倣したDSL（中間記法）を生成するだけでよく、DSL→GUI変換はツール側の静的処理として実装する。この責務分離により、AIが扱う対象がDSLに絞られるため出力のばらつきが減り、ツール側の処理は確定的になる。

### 主な利点
- **実装速度**: 従来比で大幅短縮（週単位→日単位）
- **出力安定性**: DSLに入出力が絞られることによるばらつき低減
- **コスト削減**: APIコストを従来比4〜8倍削減（出力トークンが構造化されるため）
- **配布**: サーバーレス配布が可能（変換ロジックがステートレス）
- **コンプライアンス**: AIがGUIに直接アクセスしないためセキュアな環境への導入が容易

## 他のAI統合方式との比較

| | MNP（中間DSL） | Function Calling / 構造化出力 | Computer Use（GUI直接操作） |
|---|---|---|---|
| 主な用途 | 既存GUIツールへのAI機能後付け | APIが整備された操作の呼び出し | API・DSL化できない画面の自動操作 |
| AIの出力対象 | ツール専用DSLテキスト | 事前定義された関数呼び出し | クリック・入力等の低レベル操作列 |
| 安定性 | 高い（出力空間がDSLに閉じる） | 高い（スキーマ強制） | 低い（画面変化・座標ズレに脆弱） |
| 事前実装コスト | DSL設計＋変換器の実装 | 関数・API面の整備 | ほぼ不要 |
| 操作の表現力 | DSLで定義した範囲のみ | 関数で定義した範囲のみ | 画面上の全操作 |

Function Callingとの本質的な違いは、MNPが「1回の生成で複雑な状態全体」を表現する点にある。多数の細かい関数呼び出しの連鎖になる操作（例: 図・レイアウト・構成の一括生成）はDSL 1発の方が安定し、トークンも少ない。

## 採用しないべきケース

- **操作対象のAPIが既に整備されている** — Function Callingで足りる。DSLの設計・保守という追加資産を抱える理由がない
- **内部状態が明確なデータ構造として切り出せないツール** — DSLがGUI状態を再現できないと、変換器が「解釈」を持ち始めて確定的処理という利点が崩れる
- **AIに探索的・対話的な操作をさせたい場合** — MNPは「最終状態の一括生成」に向く方式で、試行錯誤しながら画面を操作するタスクには合わない

## 実装でトレードオフが生じるケース

- **DSLの表現力 vs 生成安定性**: DSLを豊かにするほど表現できる操作は増えるが、AIの出力空間が広がり、ばらつき・文法エラーが増える。GUIの主要ユースケースに絞った最小DSLから始め、需要に応じて拡張するのが現実的
- **DSLの保守コスト**: GUI側に機能が追加されるたびDSLと変換器の更新が必要になる。GUIとDSLの二重保守を許容できるか、DSLを内部状態のシリアライズ形式と共通化して差分を減らすかの選択になる
