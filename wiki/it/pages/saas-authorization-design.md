# SaaS権限管理設計（PBAC+ReBAC二層モデル）

## 概要

マルチテナントSaaSの権限管理を「何ができるか（PBAC）」と「誰のリソースか（ReBAC）」の2層に分離して設計する手法。従来のロール直書き（`if user.role === 'admin'`）による条件分岐散在を防ぎ、ポリシークラスへの権限定義の集約とリポジトリ層でのスコープ絞り込みで認知コストの増大を構造的に抑える。スタートアップ初期はPundit相当の軽量実装から始め、規模に応じてOso Cloud・OpenFGAへ段階移行できる。

## 他の権限管理手法との比較

|  | PBAC+ReBAC二層 | RBAC（ロールベース） | OpenFGA/Zanzibar |
|---|---|---|---|
| 主な用途 | プラン×ロールが複雑なSaaS | 小規模・権限が単純なアプリ | 大規模マイクロサービス |
| マルチテナント対応 | ◎ ReBAC層でスコープ分離 | △ ロール粒度で限界 | ◎ グラフモデルで表現 |
| 権限変更の容易さ | ◎ ポリシークラスに集約 | △ 条件分岐が散在しやすい | ○ 設定ファイルで管理 |
| 導入複雑度 | 中（設計理解が必要） | 低 | 高（外部サービス依存） |
| フロントエンド共有 | ◎ API経由で権限リスト返却 | △ 個別実装になりがち | △ 別途実装が必要 |

## 採用すべきケース

- プラン×ロールの組み合わせで機能差分がある（例: フリープランは店舗3件まで）
- マルチテナント構成でテナント間データ分離が必須
- 一覧取得時にもリソースの関係性検証が必要
- フロントエンドとバックエンドで権限定義を二重管理したくない

## 採用しないべきケース

- 小規模・権限が単純（ロールが2〜3種類で十分）な場合はRBACで十分
- シングルテナント構成でテナント分離が不要な場合

## 実装のアンチパターン

### 1. ロール直書き（条件分岐の散在）

ビジネスロジック内にロール判定を直書きすると、変更時に散在した条件を全て修正する必要が生じ認知負荷が増大する。

```typescript
// Bad
if (user.role === 'admin' && user.manager && 
    user.assignee_projects.includes(projectId)) {
  // 処理
}

// Good: ポリシー呼び出しに集約
authorize({ policy: { target: 'settings', action: 'createShop' } })
```

### 2. テーブル設計で権限を表現する

`admin_leader_users` のような権限用テーブルを作るアプローチは、権限の追加・変更のたびにスキーマ変更が発生し、権限ロジックとデータ構造が密結合になる。

```sql
-- Bad: 権限ごとにテーブルを作る
CREATE TABLE admin_leader_users (user_id, project_id);
CREATE TABLE viewer_users (user_id, shop_id);

-- Good: リソーステーブルに tenant_id / shop_id を持たせ、リポジトリ層でスコープを絞る
SELECT * FROM orders WHERE tenant_id = ? AND shop_id = ?
```

### 3. フロントエンドでロールを直参照

フロントエンドがバックエンドのロール名を直接参照すると、ロール定義変更時にフロントも修正が必要になり二重管理が発生する。

```tsx
// Bad
{user.role === 'admin' && <CreateButton />}

// Good: API から返却した権限リストを参照
<Permission target="settings" action="createShop" fallback={<DisabledMessage />}>
  <CreateButton />
</Permission>
```

## 実装でトレードオフが生じるケース

### 権限管理ライブラリの選択

| 選択肢 | メリット | デメリット | 推奨場面 |
|--------|---------|-----------|---------|
| 自前実装（Pundit相当） | 依存なし・シンプル | スケールで複雑化 | スタートアップ初期 |
| Oso Cloud | 宣言的ポリシー・クラウド管理 | 外部依存・コスト | テナント数十〜百規模 |
| OpenFGA | グラフモデルで高表現力 | 導入コスト高 | 大規模マイクロサービス |

### スコープの遅延評価 vs 即時評価

- **遅延評価（推奨）**: クエリ呼び出し時にスコープを算出し、不要なメソッドではスコープ生成をスキップ。同一リクエスト内でキャッシュして二重計算を防ぐ
- **即時評価**: DI時にスコープを確定させるとテストが書きやすいが、リクエストごとの動的スコープに対応しにくい
