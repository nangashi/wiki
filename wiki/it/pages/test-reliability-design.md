# テスト信頼性設計

## 概要

テストは「実装が仕様を満たしているか」という信号を伝達するシステムである。その信号が正しくないときにのみ信頼が崩壊する。Googleの研究によれば、テストスイートの1%が信頼不能になるだけでエンジニアはテスト結果全体を信じなくなり始める。テスト信頼性設計の目的は「CIシグナルの精度を守ること」であり、（1）なぜ嘘をつくかの理解、（2）設計による構造的防止、（3）それでも出た不安定さへの運用対応、という3層で構成される。

## テストの2種類の嘘

テストが信頼を損なうパターンは2種類しかない。

### 偽陽性（False Positive）— コードは正しいのにテストが失敗する

火災報知器の誤作動に相当する。

| サブカテゴリ | 説明 |
|---|---|
| **脆いテスト** | 実装の細部（内部実装の変更）に敏感で、すぐ失敗する |
| **フレイキーテスト** | コードを変更していないのに、実行ごとに成功・失敗が変わる |

### 偽陰性（False Negative）— バグがあるのにテストが成功する

バグを検知できていない状態。

| パターン | 説明 |
|---|---|
| **空振り** | `skip` や `pending` によるテスト実行のスキップを見逃している |
| **カバレッジ不足** | テスト対象の行・分岐が網羅されていない |
| **テスト不足** | 仕様の把握が不完全で、重要なケースが抜けている |
| **自作自演** | テストコードとプロダクトコードが同じ誤った計算式を使用している |

「自作自演」は特に危険で、テストダブルを使いすぎると発生しやすい。

## テストサイズ：構造的に不安定さを防ぐ

テストの遅さと不安定さは「何をテストするか」ではなく「実行時にどこまでリソースを使うか」で決まる。Googleが提唱する実行環境の範囲による分類。

| サイズ | 定義 | 特徴 |
|---|---|---|
| **スモール** | 1プロセス内で完結 | 最速・最安定。外部I/O・ネットワークなし |
| **ミディアム** | 1マシン内で完結 | ローカルDB・ファイルシステム等は使える |
| **ラージ** | 複数マシンにまたがる | ネットワーク通信あり。最も遅く不安定 |

「ユニットテスト」と呼ばれていても、外部サービスを呼び出せばラージになる。名前ではなく実行環境の広がりがコストと安定性を決める。

低結合・高凝集の設計をすると、テストサイズを下げやすくなる。ミディアムからスモールへの移行には設計改善が必須となることが多い。

## フレイキーテストのCI分離管理

偽陽性の対処として、フレイキーテストをメインパイプラインから分離して管理するパターン。

### 分離パターン

1. フレイキーと判定したテストに `flaky` タグ（またはアノテーション）を付与
2. メインCIでは `flaky` タグのテストを除外して実行
3. 別の「フレイキーパイプライン」で `flaky` タグのみを定期実行（週次など）
4. フレイキーパイプラインの失敗はアラートにはなるが、マージブロックにはしない

この分離により「CIが落ちた = バグがある」という信号の精度を回復できる。

### フレイキーの主な原因

| 種別 | 典型例 |
|---|---|
| タイミング依存 | `time.Sleep` や固定タイムアウトに依存した非同期処理 |
| 外部依存 | 外部API・ネットワーク・DNS解決 |
| テスト間干渉 | グローバル状態・共有DBへの書き込みが後続テストに影響 |
| ランダム性 | UUIDやランダム値の順序依存 |

外部依存はテストサイズを下げること（スモール化）で構造的に排除できる。

## アサーション品質

偽陰性を減らすため、アサーションには実際の値を含める。

```python
# 悪い例: 失敗した理由が不明
assert result == expected

# 良い例: 実際の値が表示される
assert result == expected, f"got {result!r}, want {expected!r}"
```

ほとんどのテストフレームワーク（pytest, Jest, Go testing）では `assert.Equal` 系が自動的に差分を表示する。ベアな `assertTrue` / `assertFalse` は避ける。
