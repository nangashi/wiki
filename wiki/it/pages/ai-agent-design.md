# AIエージェント設計：役割先行アプローチ

## 概要

AIエージェントのチームは個別業務の列挙からでなく「役割」から設計する。役割が先に定まると業務はエージェント自身が考えて創出できる。人間のチーム設計と同型の原理であり、設計者がやるべきは「役割の定義」に絞られる。

## 詳細

### 業務先行設計のアンチパターン

やってほしい業務を1つずつ列挙していくと2つの問題が生じる：

1. **設計の限界**: 各業務の量だけでは採用・設計の判断ができない
2. **認知の限界**: AIができることについて人間が思いつける業務は限られる

「人間が業務を網羅的に想定できない」という認知限界そのものが問題であり、業務先行アプローチはこの限界を克服できない。

### 役割先行設計のパターン

役割を先に定義する（例：「営業が営業に専念できるためのオペレーションを担う」）と：

1. 役割に紐づいた業務が自然に洗い出せる
2. 採用・設計の判断軸が明確になる
3. 業務の具体化をエージェント自身に委ねられる

役割を与えられたエージェントは「この役割において何をすべきか」を自分で考えられる。人間が1つずつ考えるより、AIに業務を考えさせる方が網羅性が高い。

**人間チームとの同型性**:

```
人間チーム  : 役割 → 業務洗い出し → 採用 → 指示
AIエージェント: 役割設計 → エージェント定義 → 業務をエージェントが考える
```

### 1エージェント1役割の原則

1エージェントが複数の役割を持つとシステムプロンプトが複雑化し、品質が不安定になる。役割が混在すると「この役割において何をすべきか」という自律的判断の軸が失われる。

**悪い例**:

```
エージェントA: 顧客対応 + データ分析 + レポート作成
→ システムプロンプトが複雑化、各タスクへの集中度が下がる（経理と編集を1人に依頼してパニックになる構造と同じ）
```

**良い例**:

```
エージェントA: 顧客対応（問い合わせ受付・分類・一次回答）
エージェントB: データ分析（行動ログ集計・異常検知）
エージェントC: レポート作成（週次サマリ生成）
```
