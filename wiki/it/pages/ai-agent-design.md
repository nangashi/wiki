# AIエージェント設計：役割先行アプローチ

## 概要

AIエージェントのチームは個別業務の列挙からでなく「役割」から設計する。役割が先に定まると業務はエージェント自身が考えて創出できる。人間のチーム設計と同型の原理であり、設計者がやるべきは「役割の定義」に絞られる。

## 仕組み

### 役割先行設計のパターン

役割を先に定義する（例：「営業が営業に専念できるためのオペレーションを担う」）と：

1. 役割に紐づいた業務が自然に洗い出せる
2. 採用・設計の判断軸が明確になる
3. 業務の具体化をエージェント自身に委ねられる

役割を与えられたエージェントは「この役割において何をすべきか」を自分で考えられる。人間が1つずつ考えるより、AIに業務を考えさせる方が網羅性が高い。

**人間チームとの同型性**:

```
人間チーム  : 役割 → 業務洗い出し → 採用 → 指示
AIエージェント: 役割設計 → エージェント定義 → 業務をエージェントが考える
```

## 採用すべきケース（マルチエージェント化する条件）

役割分割＝マルチエージェント化にはコストがあるため、以下の条件を満たすタスクで採用する。

- **読み込み中心のタスク**（リサーチ・データ分析・複数情報源の並行調査） — 独立したサブタスクの並列処理でコンテキストウィンドウ制限を突破できる。Anthropicの研究タスク事例では約90%の性能向上
- サブタスク間の依存関係が弱く、暗黙の決定の共有が不要
- 単一エージェントのコンテキストに収まらない規模

## 採用しないべきケース

- **書き込み中心のタスク**（コード生成・コンテンツ作成） — サブエージェント間で暗黙の決定（スタイル・設計思想）が矛盾し、一貫性が壊れる。Cognition AIはエージェント間の完全なコンテキスト共有が極めて困難であることを根本問題として指摘している
- コンテキストウィンドウ内に問題が収まるタスク — シンプルな実装（単一エージェント・ワークフロー）から始めるのが原則。Anthropicも「ほとんどの場合Workflowsで十分」と明言している
- コスト制約が強い場合 — マルチエージェントは通常チャットの約15倍のトークンを消費する（シングルエージェントは約4倍）。性能向上の80%はトークン投入量の増加で説明できるとされる

## 実装のアンチパターン

### 1. 業務先行設計

やってほしい業務を1つずつ列挙してエージェントを設計する。

```
Bad: 「請求書処理エージェント」「メール返信エージェント」「日報作成エージェント」…
     と業務単位で追加していく
```

2つの問題が生じる: (1) 各業務の量だけでは採用・設計の判断ができない、(2) AIができることについて人間が思いつける業務は限られる。「人間が業務を網羅的に想定できない」という認知限界そのものが問題であり、業務先行アプローチはこの限界を克服できない。役割を先に定義すれば、業務の洗い出しはエージェント自身に委ねられる。

### 2. 1エージェントへの複数役割の混載

1エージェントが複数の役割を持つとシステムプロンプトが複雑化し、品質が不安定になる。役割が混在すると「この役割において何をすべきか」という自律的判断の軸が失われる。

```
Bad:
エージェントA: 顧客対応 + データ分析 + レポート作成
→ システムプロンプトが複雑化、各タスクへの集中度が下がる（経理と編集を1人に依頼してパニックになる構造と同じ）

Good:
エージェントA: 顧客対応（問い合わせ受付・分類・一次回答）
エージェントB: データ分析（行動ログ集計・異常検知）
エージェントC: レポート作成（週次サマリ生成）
```
