# IT Wiki スキーマ

IT実装事例・アンチパターン・構築事例を蓄積するwikiのルール定義。
`/ingest`・`/query`・`/lint` スキルはこのファイルを参照する。

---

## ディレクトリ構成

```
wiki/it/
  schema.md    # このファイル
  index.md     # ページ一覧
  pages/       # wikiページ（1概念1ページ）
    <slug>.md
```

---

## コンテンツタイプ

ITページは以下の2タイプに分類する。タイプによってフォーマットと品質基準が異なる。

| タイプ | 内容 | LLMフィルター |
|--------|------|-------------|
| **事例型** | 実装事例・インシデント・特定環境での問題と解決策 | 適用する |
| **リファレンス型** | 技術の全体像を網羅した参照文書（1技術1ページ） | **適用しない** |

---

## ページフォーマット

### frontmatter

```yaml
---
title: "概念名（日本語）"
type: "事例型 | リファレンス型"
sources:
  - "https://example.com/page"
  - "book:書名（著者名）"
  - "topic:トピック名"
created: "YYYY-MM-DD"
updated: "YYYY-MM-DD"
---
```

### 事例型 — 本文構成

```markdown
# 概念名

## 概要
1〜3文で概念の核心を説明する。

## 詳細
詳しい説明、背景、構成、対処法など。

## 関連
- [[関連概念スラグ]] — この概念との関係を1行で説明
- [[insight:洞察スラグ]] — 関連する洞察（insightコレクション）
```

### リファレンス型 — 本文構成（固定フォーマット）

```markdown
# 技術名

## 概要
以下の3点を含む1〜4文の連続した文章で書く。箇条書き・見出しは使わない。
1. **用途**: どの場面・領域で使われる技術か（例: 「認証・認可に使われるトークン形式」）
2. **優位性**: 代替技術と比べた最大の強み（例: 「DBを参照せずにステートレス検証できる」）
3. **核心**: 技術の仕組みや重要な注意点を簡潔に

## 他の〈領域〉技術との比較
特性軸（列）× 代替技術（行）の比較表。代替技術は2〜3個に絞る。
列は「主な用途」を必ず含め、残りは技術固有の判断軸（例: ステートレス検証・即時失効・クロスドメイン対応）を4〜5項目で構成する。

| | 〈この技術〉 | 〈代替A〉 | 〈代替B〉 |
|---|---|---|---|
| 主な用途 | ... | ... | ... |
| 〈特性軸〉 | ... | ... | ... |

## 採用すべきケース
- ケースと理由

## 採用しないべきケース
- ケースと理由

## 実装のアンチパターン
各アンチパターンにBad例コードを必須で含める。Good例も可。

### 1. アンチパターン名
説明

```lang
// Bad
...
// Good
...
```

## 実装でトレードオフが生じるケース
各ケースで選択肢・メリット・デメリット・推奨を示す。

## 関連
- [[関連概念スラグ]] — この概念との関係を1行で説明
```

## 品質基準

insight wiki の本質審査（厳格ななぜ抽出）は行わない。
代わりに以下の3観点＋LLMフィルターで評価し、**2つ以上が低品質に該当する、またはLLMフィルターに引っかかる場合はスキップ**する。

**リファレンス型はLLMフィルターを適用しない。** 3観点評価のみで判定する。

### 3観点評価

| 観点 | 低品質の兆候 | 高品質の例 |
|------|------------|-----------|
| **再現性** | 偶発的な一事例で再現条件が不明 | 「○○構成のときに△△が起きる」と条件が特定できる |
| **具体性** | ツール名・バージョン・コード・設定が一切なく抽象的 | ツール名・エラーメッセージ・コード断片が含まれる |
| **文脈** | 結論だけで背景・経緯・制約が書かれていない | 問題に至った経緯・環境・制約が記述されている |

### LLMフィルター

以下に該当する場合はスキップする:

> 「LLMが高確率で正確に答えられる、ごく一般的なIT知識である」

**スキップの目安**:
- 公式ドキュメントに明記されている基本的な設定手順
- Stack Overflow のトップ回答で広くカバーされているエラー対処
- 一般的なベストプラクティスとして広く知られているもの

**採用の目安**:
- 複数ツール・バージョン・環境が絡んだ非自明な組み合わせ問題
- なぜ一般的なアプローチが機能しないかのアンチパターン
- 特定の構成・制約下でのみ発生する問題と解決策
- 実際のインシデント・障害から得られた知見

## 共通書き込み手順

`wiki/insight/schema.md` の「共通書き込み手順」に準拠する。以下の差分を適用する:

- **ステップ1**: `wiki/it/index.md` を読む
- **ステップ2.5**: ハウツー型・観察型もLLMフィルターを通過すれば採用する
- **ステップ4**: ページは `wiki/it/pages/<slug>.md` として作成する
- **ステップ4.5**: 品質評価は上記「3観点+LLMフィルター」を使う
- **ステップ4.6**: 本質審査（サブエージェント起動）は実施しない
- **index.md の更新**: `wiki/it/index.md` を更新する
