# IT Wiki スキーマ

技術単位で情報を集約し、記事として読める体系的リファレンスを育てるwikiのルール定義。
`/ingest`・`/query`・`/lint` スキルはこのファイルを参照する。

---

## 目的

ルールの解釈に迷ったらここに立ち返る。

1. **集約**: 各所の記事・ドキュメントの情報を技術単位に集約し、あとから参照できるようにする
2. **体系的理解**: 同一技術の情報を1ページに統合し、queryだけでなく記事としてそのまま読んで技術を体系的に理解できるようにする
3. **判断支援**: トレードオフ・判断基準を載せ、状況に応じた技術選択に使えるようにする
4. **相談グラウンディング**: `/query` での技術相談時に回答の根拠となる知識ベースとして機能する

---

## ディレクトリ構成

```
wiki/it/
  schema.md    # このファイル
  index.md     # ページ一覧
  pages/       # wikiページ（1技術1ページ）
    <slug>.md
```

---

## ページ単位: 1技術1ページ

- **1ページ = 1つの技術・プロトコル・設計パターン・プラクティス領域**（例: JWT、BFF認証パターン、テスト信頼性設計）
- 新しい情報は**既存ページへの統合を最優先**する（集約ファースト）。新規ページを作るのは対象技術のページが存在しない場合のみ
- ページが肥大化したら、独立性の高いサブトピックを子ページに分割し、親ページからリンクする（目安: 1つのサブトピックだけでページの過半を占めるようになったら分割候補）
- スラグは全コレクション横断でユニーク（`wiki/collections.md` 参照）

---

## コンテンツタイプ

| タイプ | 位置づけ | 内容 |
|--------|---------|------|
| **リファレンス型** | **標準** | 技術の全体像・判断基準・実践知を集約した参照文書（1技術1ページ） |
| **事例型** | 例外 | どのリファレンスページにも収まらない独立した事例・インシデント |

個別の実装事例・インシデント・運用知見は、原則として該当技術のリファレンスページ（「事例・運用知見」「実装のアンチパターン」等のセクション）に統合する。
事例型ページを新規に作るのは、複数技術にまたがる等でどのリファレンスページにも自然に収まらない場合のみとし、**作成前にユーザーに確認する**。

---

## ページフォーマット

### frontmatter

```yaml
---
title: "技術名（日本語）"
type: "リファレンス型 | 事例型"
sources:
  - "https://example.com/page"
  - "book:書名（著者名）"
  - "topic:トピック名"
created: "YYYY-MM-DD"
updated: "YYYY-MM-DD"
---
```

### リファレンス型 — 本文構成

必須は「概要」「関連」のみ。他のセクションは該当する内容がある場合に置く。
複数ソースから漸進的に育てる前提のため、**初版で全セクションを埋める必要はない**。ただし判断系セクション（比較・採用すべき/しないケース・アンチパターン・トレードオフ）が1つもないページは情報が薄い状態とみなす（品質基準参照）。

```markdown
# 技術名

## 概要 【必須】
以下の3点を含む1〜4文の連続した文章で書く。箇条書き・見出しは使わない。
1. 用途: どの場面・領域で使われる技術か
2. 優位性: 代替技術と比べた最大の強み
3. 核心: 技術の仕組みや重要な注意点を簡潔に

## 仕組み
体系的理解のためのセクション。構成要素・動作原理・全体の構造を書く。
LLMが答えられる一般知識でも、記事として自立して読むのに必要なら書いてよい。

## 他の〈領域〉技術との比較
特性軸（列）× 代替技術（行）の比較表。代替技術は2〜3個に絞る。
列は「主な用途」を必ず含め、残りは技術固有の判断軸を4〜5項目で構成する。

| | 〈この技術〉 | 〈代替A〉 | 〈代替B〉 |
|---|---|---|---|
| 主な用途 | ... | ... | ... |
| 〈特性軸〉 | ... | ... | ... |

## 採用すべきケース
- ケースと理由

## 採用しないべきケース
- ケースと理由

## 実装のアンチパターン
各アンチパターンにBad例コードを必須で含める。Good例も可。

### 1. アンチパターン名
説明

## 実装でトレードオフが生じるケース
各ケースで選択肢・メリット・デメリット・推奨を示す。

## 事例・運用知見
個別の実装事例・インシデント・ベンチマーク等の吸収先。
環境・バージョン・制約条件を明記する（どの条件で再現する知見かが分かること）。

## 関連 【必須】
- [[関連技術スラグ]] — この技術との関係を1行で説明
- [[insight:洞察スラグ]] — 関連する洞察（insightコレクション）
```

### 事例型 — 本文構成

```markdown
# 事例名

## 概要
1〜3文で事例の核心を説明する。どの環境・条件で起きた何か、が分かること。

## 詳細
経緯・環境・原因・対処。再現条件（構成・バージョン・制約）を必ず含める。

## 関連
- [[関連技術スラグ]] — この事例との関係を1行で説明
```

---

## 命名規則

- **ファイル名**: 英語またはローマ字表記の kebab-case（例: `jwt.md`, `bff-auth-pattern.md`）
- **タイトル**: 日本語（frontmatter の `title` フィールドおよび本文 `# 見出し`）
- **スラグ**: ファイル名から `.md` を除いたもの

---

## 採用基準

情報を取り込むかどうかは「**ページのどのセクションを充実させるか**」で判断する（セクション貢献基準）。

**採用**:
- 比較・採用判断・アンチパターン・トレードオフ・事例運用知見のいずれかに寄与する
- 概要・仕組みの体系的理解に必要（一般的な知識でも、記事として自立して読むのに必要なら採用する）

**スキップ**:
- どのセクションにも寄与しない（例: 公式チュートリアルの逐次手順の転記、特定バージョンの設定オプションの網羅的一覧）
- 既存ページに同内容が記録済み

### 事例型ページの追加基準（3観点）

独立の事例型ページを作る場合のみ、以下の3観点で評価し、**2つ以上が低品質に該当する場合はスキップ**する。

| 観点 | 低品質の兆候 | 高品質の例 |
|------|------------|-----------|
| **再現性** | 偶発的な一事例で再現条件が不明 | 「○○構成のときに△△が起きる」と条件が特定できる |
| **具体性** | ツール名・バージョン・コード・設定が一切なく抽象的 | ツール名・エラーメッセージ・コード断片が含まれる |
| **文脈** | 結論だけで背景・経緯・制約が書かれていない | 問題に至った経緯・環境・制約が記述されている |

## 共通書き込み手順

`/ingest`・`/query` の両スキルで使用する。

### ステップ1: 既存ページを把握する
`wiki/it/index.md` を読み、すでに存在する技術ページとその内容を把握する。

### ステップ2: 対象の技術・パターンを特定する
ソースが扱う技術・パターンを列挙する。

- 1つのソースが複数の技術に触れる場合は技術ごとに分割する
- 記事の主題だけでなく、途中で語られる別技術の判断材料（比較・失敗談・ベンチマーク等）も取り込み対象になりうる

### ステップ3: 技術ごとに統合先を判断する

```
対象技術の既存ページがある？
  YES → 既存ページに統合（追記）
  NO  → 1技術1ページの単位として独立したページを作る価値がある？
    YES → 新規リファレンスページを作成
    NO  → 最も関連する技術ページの該当セクションに追記。
          どのページにも収まらない場合のみ事例型ページを提案（ユーザー確認）
```

### ステップ4: ページを書く・更新する

**追記の場合（標準ルート）:**
- 新しい情報を該当セクションにマージする。セクションがなければフォーマットの順序に従って追加する
- **矛盾チェック**: 既存の記述と矛盾する場合、(a) 条件・環境が違うなら条件を明記して併記、(b) 既存情報が古い・誤りなら更新。判断できない場合はユーザーに確認する
- 追記後にセクション全体を読み直し、時系列の継ぎ足しではなく1つの記事として流れるように整える（「なお」「また」の連鎖は書き直しの兆候）
- `updated` を今日の日付に更新し、`sources` に新しいソースを追加する

**新規の場合:**
- 命名規則に従い `wiki/it/pages/<slug>.md` を作成
- frontmatter: title・type・sources・created・updated を記入
- 本文: リファレンス型フォーマットに従う（埋められるセクションだけでよい）

### ステップ5: 品質確認
下記「品質基準」の2条件を確認する。未達の場合は補足調査を提案する:

```
### 品質確認: [[スラグ]] — タイトル

以下の点で情報が薄い状態です:
- （未達の条件とその根拠）

補足調査を行いますか？（キーワード案: 「...」「...」）
```

- ユーザーが承認した場合: その場で `WebSearch` を実行し、ページを更新してから次のステップへ進む
- ユーザーが断った場合: そのまま保持し、ステップ8のサマリに「補足調査未実施」として記録する
- 品質が十分な場合: このステップは何も出力しない

### ステップ6: 関連リンクを相互に張る
- 新規・更新ページの「関連」セクションに関連技術へのリンクを追加
- 関連先ページの「関連」セクションにも逆リンクを追加
- insightコレクションに関連する洞察があれば `[[insight:slug]]` で参照する

### ステップ7: index.md を更新する
新規ページを作成した場合、または既存ページの概要が変わった場合は `wiki/it/index.md` を更新する。
```
- [[slug]] — 1〜2文のサマリ
```

### ステップ8: 作業サマリを出力する
以下を報告する:
- 新規作成したページ一覧（タイトル・スラグ）
- 更新したページ一覧（タイトル・追記セクション・内容の要約）
- スキップした情報（理由付き）
- 追加した関連リンク
- 補足調査未実施のページ（ユーザーが断った場合のみ）

---

## 品質基準

リファレンス型ページは以下の2条件で評価する。

1. **判断に使える**: 比較・採用すべき/しないケース・アンチパターン・トレードオフのいずれか1つ以上が実質的な内容を持つ（見出しだけ・1行だけは不可）
2. **記事として自立**: 概要（＋仕組み）だけで技術の全体像が掴め、ソースを直接参照しなくても読める

insight wiki の本質審査（サブエージェントによる審査）は行わない。

## ソース記録ルール

frontmatter の `sources` に記録する形式:

| 取込元 | sources の記述形式 |
|--------|------------------|
| 特定のWebページ | `"https://..."` |
| 特定のPDF | `"file:ファイル名.pdf"` |
| 本に関するWeb収集 | `"book:書名（著者名）"` + 収集したURL群 |
| トピックに関するWeb収集 | `"topic:トピック名"` + 収集したURL群 |

Web収集の場合、特殊マーカー（`book:` / `topic:`）を **先頭** に置き、その後に実際に参照したURLを列挙する。
集約によりソースが増えていくため、追記のたびに新しいソースを追加する（重複は追加しない）。
