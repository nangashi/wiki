# /ingest スキル

指定したソースを読み込み、wikiページを作成・更新する。

## 事前準備

`wiki/schema.md` を読み、wikiのルールと共通書き込み手順を確認する。

---

## 採用基準

ソースから抽出した情報のうち、以下の基準を満たすものだけをwikiページとして記録する。満たさない情報はスキップする。

### 基本原則

**具体的な情報より原理原則・insightを優先する。**
具体的な情報（ツールの使い方、技法の手順）はすぐ陳腐化するが、原理原則が変わることは少ない。

### 採用フィルター（すべてを満たすことが望ましい）

1. **抽象度が高い** — 特定ツール・特定バージョン・特定言語に依存せず、より広い文脈に適用できる
2. **陳腐化しにくい** — ツールのアップデートや仕様変更で内容が無効にならない
3. **LLMが深く理解していない** — 一般的なプログラミング・設計知識としてLLMがすでに標準的な回答を持っている内容は記録する価値が低い
4. **新規性・発見性がある** — 誰かが新たに見つけてきた観察・フレームワーク・反直感的な知見である

### 例外：具体的な情報でも採用してよい条件

上記フィルターを完全には満たさなくても、**「LLMに渡すことで回答品質が一定程度上がる」** 場合は採用する:

- LLMが個々の要素は知っていても、**その組み合わせ・適用パターン**を自発的に提案しにくいもの
- LLMのデフォルト回答とは**異なる方向性**（反直感的・反通説的）を示すもの

---

## 引数フォーマットとソース種別

引数のパターンから自動的にソース種別を判定する。

| 引数の形式 | ソース種別 |
|-----------|-----------|
| `https://...` または `http://...` | Webページ |
| `.pdf` で終わるパス | PDF |
| `本:` または `book:` で始まる文字列 | 本に関するWeb収集 |
| `トピック:` または `topic:` で始まる文字列 | トピックに関するWeb収集 |

引数が複数ある場合は順に処理する。

---

## ソース種別ごとの処理

### Webページ

1. `WebFetch` で指定URLの内容を取得する
2. ページの主要な概念・情報を抽出する
3. sources frontmatter: `["<URL>"]`

### PDF

1. `Read` ツールで指定PDFファイルを読み込む
2. ドキュメントの主要な概念・情報を抽出する
3. sources frontmatter: `["file:<ファイル名>"]`

### 本に関するWeb収集

書名（著者）を引数として受け取る。例: `本: リファクタリング（Martin Fowler）`

1. 以下の観点で `WebSearch` を複数回実行する（計5〜8回）:
   - 書籍の概要・目次
   - 主要概念・重要章のサマリ
   - レビュー・解説記事
   - 著者の他の著作との比較・関連
2. 収集したURLをすべてリストアップする
3. 取得した情報から主要概念を抽出する
4. sources frontmatter: `["book:書名（著者名）", "https://url1", "https://url2", ...]`
   - `book:` マーカーを先頭に置く

### トピックに関するWeb収集

トピック名を引数として受け取る。例: `トピック: 強化学習`

1. 以下の観点で `WebSearch` を複数回実行する（計5〜8回）:
   - トピックの定義・概要
   - 主要な技術・手法・概念
   - 代表的な論文・事例
   - 最近の動向・課題
2. 収集したURLをすべてリストアップする
3. 取得した情報から主要概念を抽出する
4. sources frontmatter: `["topic:トピック名", "https://url1", "https://url2", ...]`
   - `topic:` マーカーを先頭に置く

---

## wikiへの書き込み

ソース種別に関係なく、`wiki/schema.md` の **「共通書き込み手順」** に従ってwikiページを作成・更新する。

---

## 出力形式

処理完了後、以下を報告する:

```
## 取込完了

**ソース**: [ソース種別] <入力値>

### 新規作成 (N件)
- [[スラグ]] — タイトル

### 更新 (N件)
- [[スラグ]] — タイトル（追加内容の要約）

### スキップ (N件)
- タイトル — 理由（例: 既に詳細に記録済み、ハウツー型・原理なし）

### 追加したリンク
- [[A]] ↔ [[B]]: 関係の説明

### 本質審査
- [[スラグ]] — 合格 / 改善（改善した基準と内容の要約）
```
