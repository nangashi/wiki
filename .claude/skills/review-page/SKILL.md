# /review-page スキル

既存のwikiページを見直し、品質改善を行う。

`/ingest` の本質審査（essence-review-criteria）を遡及適用するだけでなく、**読者の有用性**（理解しやすさ・行動移転性）と **wiki 構造**（リンク品質・統合判断）も評価する。

Codex を使った深掘りレビューは `--codex` フラグを付けると利用できる（後述）。

---

## 引数

```
/review-page <slug>              # 1ページを見直す
/review-page <slug1> <slug2> ... # 複数ページを見直す
/review-page --all               # 全ページをスキャン（reviewed なし or 古いバージョンを優先）
```

コレクション指定が必要な場合は `insight:<slug>` または `it:<slug>` の形式を使う。

---

## 事前準備

1. `wiki/collections.md` を読み、コレクション構成を確認する
2. 対象コレクションの `schema.md` を読む
3. `wiki/insight/references/essence-review-criteria.md` を読む（insight コレクションの場合）
4. `wiki/insight/index.md` と `wiki/it/index.md` を読み、関連ページのサマリを把握する

---

## レビュー基準

### R1: 理解しやすさ（Comprehensibility）

ページを読むだけで内容が理解できるか。

**合格の条件**:
- 前提知識となる概念がリンクまたは説明で補われている
- 詳細セクションが概要の言い換えでなく、新情報を持つ
- 抽象的な主張について、理解に必要な場合は具体例・条件・反例・ソース上の根拠が補われている

**不合格の兆候**:
- 用語が定義なく使われており、ソースを参照しないと分からない
- 詳細が概要の膨らみに過ぎず、読み進めても新しい知識が得られない
- 具体例がなく「〜という傾向がある」という抽象記述のみ（理解に必要な場合）

**改善アクション**: 抽象主張に必要最小限の具体例・条件・反例を追加する / 詳細セクションに概要から導けない新情報を補充する / 核心に寄与しない例は削る

---

### R2: 行動移転性・運用移転性（Actionability）

insight では「このページを読んだ後、認識・判断・行動をどう変えるか」が分かるか。  
it では「どの条件で使える知識か、再現・判断・回避にどう使うか」が分かるか。

**合格の条件（insight）**: 概要または詳細を読み終えたとき「これをどう使うか」が導ける。知識の記述ではなく、認識・判断・行動の変化が想起できる。

**合格の条件（it）**: 再現条件・環境・制約が分かる（事例型）、またはどのケースで採用すべきか・アンチパターン・トレードオフが判断に使える（リファレンス型）。

**不合格の兆候（insight）**:
- 「〜という現象がある」で終わっており、「だから〜する/しない」が導けない
- 認知バイアス・心理効果を扱うページで対処法が書かれていない

**不合格の兆候（it）**:
- 事例の経緯のみで、再現条件・原因・対処が分からない
- 技術の説明のみで、採用判断・アンチパターン・トレードオフが書かれていない

**改善アクション**: 「実践への示唆」または「対処」セクションを追加する。対処の根拠は核心から導くこと（essence-review-criteria C3 と連動、insight のみ）

---

### R3: 本質の核心性・事実基盤 [insight コレクション必須]

`wiki/insight/references/essence-review-criteria.md` の現行バージョンを遡及適用する。

**必須判定**:
- C1: 核心の一文性
- C2: 核心から現象が逆算できる
- C3: 対処が本質への干渉として説明されている（対処・実践セクションがある場合）
- C6: 分類の根拠（分類・タクソノミー型の場合）
- C5: 詳細セクションの言語・ツール非依存性（ソースが言語/ツール固有の場合）
- S1: ソース格付け
- S2: 事実主張の検証（数値・実験結果・固有名を伴う主張がある場合）

**任意判定**:
- C4: 適用限界（誤用リスクが高い原理では改善提案に含める）

**スキップ条件**: `reviewed` フィールドが essence-review-criteria の現行バージョンと一致する場合はスキップ可能。ただし、本文・sources・対処セクション・数値主張が変更されている場合は再審査する。

---

### R4: リンク品質

**不足（リンク漏れ）**:
- 本文中で言及されている wiki 概念に `[[link]]` がない
- 前提・上位概念として参照すべきページがリンクされていない

**過剰（ノイズ）**:
- 関係が薄い概念へのリンクが「関連」セクションに並んでいる
- リンクはあるが関係の説明（1行コメント）が書かれていない

**方向性**:
- 上位概念 ↔ 下位概念・対比・強い依存関係がある場合のみ相互リンクを推奨する

**判定方法**:
1. 「関連」セクションの各リンク先ページを読み、関係の強さを評価する
2. 本文中の概念名を `wiki/insight/index.md` と `wiki/it/index.md` の両方と照合し、リンク漏れを検出する
3. 同一コレクションは `[[slug]]`、別コレクションは `[[insight:slug]]` / `[[it:slug]]` を使う

---

### R5: 統合・分割の是非

**統合候補（マージ）**:
- このページの内容が、別の関連ページのサブセクションとして収まる
- 他ページと内容が重複しており独自情報が少ない
- 独立した概念として読者に価値を提供できていない

**分割候補**:
- 複数の独立した概念を同列に説明している
- 「関連」セクションが 10 個以上（複数概念の兆候）

**判定方法**: `wiki/[collection]/index.md` を読み、内容的に近いページと照合する。

**注意**:
- 統合・削除・リダイレクト化は破壊的変更なので、必ずユーザー確認を取る
- 被参照が多いページ・独自の検索入口になっているページは保持を優先する
- 関係概念ページとして成立する場合は、単純マージではなく「上位の関係概念ページ作成」を提案する

---

## 実行手順

### ステップ1: 対象ページを読む

対象ページを読み、`reviewed` フィールドを確認する。

### ステップ2: 各基準を評価する

R1〜R5 を順に評価する。R3 は `reviewed` が現行バージョンと一致する場合スキップ可。

### ステップ3: 結果をサマリ形式で提示する

改善提案は優先度順に並べる:
1. 事実誤り・ソース不足（R3 の S1/S2）
2. 本質審査不合格（R3 の C1/C2/C3）
3. 行動移転性・運用移転性の不足（R2）
4. 理解しやすさの問題（R1）
5. リンク漏れ・リンクノイズ（R4）
6. 統合・分割・削除候補（R5）

### ステップ4: 改善を対話処理する

問題ごとに:
1. 何が問題か・推奨修正を提示する
2. 「この修正を適用しますか？」と確認する
3. 承認があれば Edit ツールでファイルを更新する

軽微な文言修正・リンク追加はまとめて確認してよい。  
統合・分割・削除・リダイレクト化・大幅な核心文変更は 1 件ずつ確認する。

全件処理後、`updated` フィールドを更新する。  
insight ページで R3 の現行必須基準すべてに合格した場合のみ、`reviewed` を現行バージョンに更新する。  
index.md の要約に影響する変更を行った場合は、対応する `wiki/[collection]/index.md` も更新する。

```
## レビュー結果: [[slug]] — タイトル

| 基準 | 判定 | 詳細 |
|------|------|------|
| R1: 理解しやすさ | ✓合格 / ✗不合格 | ... |
| R2: 行動移転性・運用移転性 | ✓合格 / ✗不合格 | ... |
| R3: 本質の核心性・事実基盤 | ✓合格 / ✗不合格 / スキップ / — (it) | 核心文: "..." |

1. [R3-S2] ...
2. [R2] ...
```

## Codex 協調モード（`--codex` フラグ）

`/review-page --codex <slug>` で Codex を使った深掘りレビューを実行する。

**利用推奨条件**（いずれかに該当する場合に使う）:
- sources が弱い・古い・二次資料のみ
- 数値・研究名・固有名を含む主張がある
- 最近変化しうる技術・標準・製品を扱う
- ページ内容に反論・限界条件が不足している

**通常レビューで十分な場合**:
- wiki 内のリンク整理・統合判断・概要と詳細の重複整理

### Phase 1: Codex（トピック独立調査・外部検証）

`codex:rescue` スキルを使って以下のプロンプトで Codex を起動する。`{タイトル}`・`{概要}` を実際の値に置換すること。

```
以下のwikiページについて、独立した立場からレビューを行ってください。

## ページ情報
タイトル: {タイトル}
概要: {概要}

## タスク

1. **トピック独立調査**: WebSearch でこのトピックを調査する（3〜5回）。一次資料・公式資料・原論文を優先し、必要に応じて二次資料で文脈を補う。ページを書いた人とは独立に調査すること。

2. **ギャップ分析**: ページが扱っていない重要な側面・反論・限界条件を列挙する。

3. **事実精度**: ページの主張（数値・研究名・定義）の正確性を評価する。確認できたものと疑わしいものを分けること。

4. **行動指針の欠落**: このトピックで「実践者が知るべき最重要の行動指針」は何か。ページはそれをカバーしているか。

5. **適用限界**: この主張が成り立たない条件・反論・誤用パターンは何か。

## 出力形式

### 重要なギャップ
- （ページが扱っていない重要な観点）

### 事実精度
- 確認できた主張: ...
- 疑わしい主張: ...

### 行動指針の欠落
- （ページが触れていない重要な実践知）

### 適用限界
- （誤用パターン・反論・成立しない条件）

### 補足情報
- （ページを補強できる追加情報・出典）
```

Codex レポートを受け取ったらその内容を保持したまま Phase 2 に進む。

### Phase 2: Claude Code（レビューと改善）

Codex レポートをインプットに加えて R1〜R5 を評価し、Codex が指摘したギャップ・精度問題も改善提案に含める。改善の実施は通常モードのステップ4と同様に対話処理する。

---

## 注意事項

- `--all` は対象ページが多い場合、時間がかかる。`reviewed` なしのページを優先することを推奨する
- R3（本質の核心性・事実基盤）は insight コレクションのみ適用する
- 核心に寄与しない説明・具体例は「有用な情報に絞る」方針に従い、改善時に削除してよい
