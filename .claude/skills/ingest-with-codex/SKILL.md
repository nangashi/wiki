# /ingest-with-codex スキル

Codexとの3段階パイプラインでwikiページを作成・更新する。

- **Phase 1**: Codex が一次資料収集・骨格設計を行う
- **Phase 2**: Claude Code が Codex レポートを基に拡張調査し、wiki に書き込む
- **Phase 3**: Codex が草稿をファクトチェックし、未確認主張を排除する

Codex なし（Claude Code 単体）で取り込む場合は `/ingest` を使用すること。

**重要**: Codex は worktree を使わず、本体ディレクトリで実行する。

---

## 事前準備

1. `wiki/collections.md` を読み、コレクション構成と判定ルールを確認する
2. コレクション判定（後述）を実行してユーザーに確認を取る
3. 対象コレクションの `schema.md` を読み、ルールと共通書き込み手順を確認する

---

## コレクション判定

ソースの内容（URL・タイトル・概要）から、どのコレクションに追加するかを判定する。

`wiki/collections.md` の「判定目安」に照らして各コレクションへの該当可否を評価し、以下のフォーマットでユーザーに提示して確認を取る:

```
**コレクション判定**

- insight: [YES / NO / 候補] — 理由
- it: [YES / NO / 候補] — 理由

→ [判定コレクション] に追加しますか？
```

ユーザーが承認したコレクションに対してのみ以降の処理を実行する。複数コレクションへの追加も可。

---

## 引数フォーマットとソース種別

引数のパターンから自動的にソース種別を判定する。

| 引数の形式 | ソース種別 |
|-----------|-----------|
| `https://...` または `http://...` | Webページ |
| `.pdf` で終わるパス | PDF |
| `本:` または `book:` で始まる文字列 | 本に関するWeb収集 |
| `トピック:` または `topic:` で始まる文字列 | トピックに関するWeb収集 |

引数が複数ある場合は順に処理する。

## Phase 1: Codex（一次資料収集・骨格）

`codex:rescue` スキルを使って以下のプロンプトでCodexを起動する。
`{トピック名}` と `{ソース種別}` を実際の値に置換すること。

```
トピック「{トピック名}」（ソース種別: {ソース種別}）について以下を実行してください。

## タスク

1. **一次資料収集**: WebSearchとWebFetchを使って論文・原典・信頼できる資料に当たる（5〜8回の検索）。二次資料（解説ブログ等）より一次資料（原論文・原典PDF・公式資料）を優先すること。

2. **核心文の確定**: このトピックの本質を、一読で理解できる簡潔な1文（目安: 読点2〜3個・80字以内）で表現する。圧縮で失われるニュアンスは直後の補足1文で補ってよい。使用したフレームとその根拠（どの一次資料に基づくか）を明示すること。

3. **骨格設計**: wikiページの節構成（見出しのみ）を提案する。

4. **弱点の特定**: 自分の調査で確認できなかった・補強が必要な点を列挙する。Claude Codeへの補完調査依頼として記述すること。

## 出力形式（必ず遵守）

### 核心文
（1〜2文）

### フレーム根拠
（なぜそのフレームが正確か、どの資料に基づくか）

### 骨格
- 節名: 説明
（全節を列挙）

### 弱点・要補強点
- （Claude Codeに追加調査させる具体的な項目）

### 参照した一次資料
- （URL・文書名・確認内容）
```

Codexのレポートを受け取ったら、その内容を保持したままPhase 2に進む。

---

## Phase 2: Claude Code（拡張調査・肉付け・wiki書き込み）

Phase 1のCodexレポートを入力として、以下を実行する。

1. **フレームの採用**: Codexの核心文とフレームを基準として採用する（Codexの一次資料調査が基盤のため）
2. **弱点の補完調査**: Codexが「弱点・要補強点」に列挙した項目を `WebSearch` で補完調査する
3. **拡張**: Codexの骨格に以下を追加する
   - 関連研究・事例・数値
   - 詳細な説明
   - 関連するwikiページへのリンク
4. **wiki書き込み**: 対象コレクションの `schema.md` の書き込み手順に従ってページを作成する

**Phase 2のみがwikiファイルに書き込む。**

草稿が完成したらその内容を保持したままPhase 3に進む。

---

## Phase 3: Codex（ファクトチェック）

`codex:rescue` スキルを使って以下のプロンプトでCodexを起動する。
`{草稿内容}` を実際のwikiページ内容に置換すること。

```
以下のwikiページ草稿をファクトチェックしてください。

## チェック方針

1. 強い主張（断定形・統計数値・研究名の引用）を特定する
2. 一次資料で確認できないものを「未確認」と分類する
3. 未確認の主張には修正案（表現の弱化・削除・出典要求のいずれか）を提示する

## 草稿

{草稿内容}

## 出力形式

### 確認済み主張
- 「引用文」: 根拠となる資料

### 未確認主張
- 「引用文」: 修正案

### 総評
（1〜2文）
```

Codexのファクトチェック結果を受け取ったら:
- 「未確認主張」に挙げられた箇所を修正する（表現の弱化・削除・出典追記）
- 修正内容が実質的な変更を伴う場合はユーザーに差分を提示する
- 修正により概要の核心文が変わった場合は `index.md` のサマリ（1文目＝核心文）も更新する

---

## 出力形式

処理完了後、以下を報告する:

```
## 取込完了（協調モード）

**ソース**: [ソース種別] <入力値>

### [コレクション名] — 新規作成 (N件)
- [[スラグ]] — タイトル

### [コレクション名] — 更新 (N件)
- [[スラグ]] — タイトル（追加内容の要約）

### スキップ (N件)
- タイトル — 理由

### 追加したリンク
- [[A]] ↔ [[B]]: 関係の説明

### 本質審査（insightコレクションのみ）
- [[スラグ]] — 合格 / 改善（改善した基準と内容の要約）

### 補足調査未実施（該当時のみ）
- [[スラグ]] — 未実施の観点（ユーザーが補足調査を断った場合）

### ファクトチェック結果
- 修正した主張: N件
- 修正内容の要約
```
