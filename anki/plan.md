# Anki フラッシュカード生成計画

insight wiki のページから、概念・考え方の習得を目的とした Anki フラッシュカードを生成するスキルを作る計画。

## このファイルの使い方（エージェントへの指示）

このplanを見たエージェントは以下の手順で作業すること:

1. **まず「未調査事項」セクションを確認する**。未調査の項目が残っていれば、調査を実施して結果を各項目の直下に「調査結果:」として追記し、ステータスを `[x]` に更新する
2. 調査結果を踏まえて「設計方針」「カード生成基準」の暫定内容を修正・具体化する（変更点は明記する）
3. 「要ユーザー確認」の項目は勝手に決めず、選択肢と推奨案を整理してユーザーに提示する
4. 全未調査事項が解消し、要ユーザー確認事項が決着したら、「実装タスク」に沿って実装に進む
5. planと実態が乖離したら planを直すこと（planが正、コードが従ではなく、両方を同期させる）

## 目的・スコープ

- **目的**: insight wiki の概念・考え方を長期記憶に定着させる（暗記ではなく理解の想起）
- **対象**: `wiki/insight/pages/` のみ（現在72ページ）。it コレクションは対象外
- **成果物**: `/anki` スキル（`.claude/skills/anki/`）。単発のカード生成だけでなく、**insight wiki と Anki の継続的な同期**を実現する: (1) カード生成時に生成元・生成結果を記録する (2) 記事の追加・変更・削除を検知したらカードへの反映案を提案する (3) ユーザー承認後にAnkiへ反映する
- **言語**: カードは日本語（wikiのタイトル・本文が日本語のため）

## 設計方針（暫定）

### カードの内容方針（調査2・3・4を反映、2026-07-19）

insight wiki のスキーマ（`wiki/insight/schema.md`）はページを型分類しており、カード生成もこれを利用できる見込み:

- **原理・原則型**（認知バイアス等）: 「なぜ起きるか（メカニズム）」「どう対処するか」「適用トリガー（どんな状況で思い出すべきか）」の3方向で問える
- **理論・フレームワーク型**: 「提唱動機・解こうとした問題は何か」を問う
- **パターン・実践型**: 「このパターンはどんな問題を解くか」「なぜ有効か」を問う
- **対比・関係概念**: 「XとYの違いは何か」を問う

各ページの「概要」セクション1文目は本質審査を経た核心文なので、カードの答え側の第一候補になる。ただし、文をそのまま暗記させるのではなく、問いに必要な最小単位へ分解する。

- **理解してから記憶する**: ページの核心・因果関係を解釈できない場合はカード化せず、ページ改善候補として報告する
- **Basicを標準とする**: 「概念→内容を説明」「事象→概念を特定」「事象→対処を導出」の3方向を基本とする。メカニズムの個別要素を別カードへ過度に断片化しない
- **Clozeは限定する**: 文脈を保ったまま単一の関係・構成要素を想起させる場合だけ使う。概念名だけを穴にする、長文の複数箇所を同時に隠す、穴以外から答えを推測できるものは作らない

### 想起方向の方針

概念習得が目標なので、単純な「用語→定義」だけでなく双方向・状況起点を含める:

1. **概念→内容説明**: 「アンカリング効果とはどのような現象か。なぜ起きるか？」→ 現象と核心メカニズムを一まとまりで説明する
2. **事象→概念**: 「交渉で相手が先に極端な数字を出し、合意額がそこへ引き寄せられた。何が起きているか？」→ アンカリング効果。適用トリガーを素材にする
3. **事象→対処**: 「交渉相手から最初の金額を提示されそうだ。判断の歪みを減らすには？」→ 概念を想起し、アンカーを見る前に独立した基準を決める
4. **対比**（対比自体が概念の核心である場合）: 「依存・自立・相互依存の違いを、関係の選択可能性から説明すると？」

事象→対処カードの答えには、対処だけでなく根拠となる**概念名を明記する**。これにより、復習後に概念名からwikiや外部資料を検索できるようにする。

### 品質基準（調査2・4を反映、2026-07-19）

- カード化の前提知識を先に理解している（SuperMemoの「理解してから記憶する」）
- 1カード1概念または1判断。minimum information principleは「概念を構成する意味上不可分な因果まで分断する」ことではなく、独立した複数概念を詰め込まない基準として使う
- 概念説明カードでは、現象・核心メカニズム・実用上重要な含意を一まとまりで答えてよい。個々のメカニズムや例を別カードへ機械的に分解しない
- 答えは原則1〜3文。列挙の完全再現ではなく、意味が説明できたかで採点できる形にする
- 問題文だけで何を問われているか一意に決まる（文脈依存の「これは何？」を作らない）
- 一覧の丸暗記、列挙数を要求する問い、Yes/No問題、表現だけを暗記する問題は避ける
- 似た概念との干渉が予想される場合は、違いを明示した対比カードにする
- 状況問題はwikiに根拠のある適用トリガーから作る。LLMがもっともらしい事例を創作しない
- 各カードに出典ページのスラグをタグまたは補足欄で記録し、wikiに戻れるようにする
- ページあたりのカード数目安: 1〜3枚。原則は「概念→内容説明」1枚＋有用な適用トリガーがある場合だけ「事象→概念」「事象→対処」を追加する

### ノートタイプ・デッキ方針（調査3を反映、2026-07-19）

- ノートタイプは専用の `Insight Basic` を基本とし、フィールドは `Question`、`Answer`、`Extra`、`Source`、`CardId`。`CardId` は表示しない
- Clozeを採用する場合だけ `Insight Cloze` を別に作り、`Text`、`Extra`、`Source`、`CardId` を持たせる
- デッキは原則1つ（`insight`）。カテゴリは `insight::説得・影響力` のような階層タグで表し、復習キューを分断しない
- サブデッキは、カテゴリごとに学習上限・オプション・学習時期を意図的に変える必要がある場合だけ採用する

### 再生成・更新の方針（調査1・4・5を反映、2026-07-19）

- カードには安定した決定論的IDを持たせる。`<slug>::<想起方向>::<意味上の連番>` を基にし、文面変更ではIDを変えない
- IDをカスタムフィールド `CardId` と `anki/state.json` の両方に保存する。AnkiConnectでは `CardId` を検索して既存note IDを得て更新する
- `.apkg` では同じ入力から常に同じGUIDを生成する。GUID・モデルID・フィールド構成を変更すると更新が壊れ得るため固定する
- `state.json` にはページの `updated` だけでなくコンテンツハッシュ、生成基準バージョン、カードID、投入先note ID（AnkiConnect時）を記録する。日付だけに依存しない
- 自動削除は初期スコープ外。生成元から消えたカードは `stale` として提示し、ユーザー承認後に停止または削除する

### 同期ワークフロー（2026-07-19 追記・実装済み）

wikiを正本、Ankiを派生物とする一方向同期。3つの動作で構成する。

**1. 記録（実装済み）**
カードをAnkiへ投入したら `anki/state.json` にページのコンテンツハッシュ・基準version・CardId・note IDを記録する。この記録が検知の基準点になる。

**2. 検知と提案（実装済み）**
`/anki` を引数なしで実行したら sync モードとして動作する: `wiki/insight/pages/*.md` と `state.json` を突合し、全ページを次の4分類に仕分けて報告する。

| 分類 | 判定 | 提案内容 |
|------|------|---------|
| 新規 | stateにエントリがないページ | カード案を生成して提示 |
| 変更 | contentHashが不一致、または criteriaVersion が現行と不一致 | 下記「変更ページの反映提案」 |
| 削除 | stateにあるが pages/ にないページ | 所属カード全件の stale 化を提示 |
| 未変更 | ハッシュ・versionとも一致 | 件数のみ報告 |

**変更ページの反映提案**: ページの現内容から改めてカード案を作り、既存カードとCardIdで突合して、カード単位のアクション案（維持 / 文面更新 / stale化 / 新規追加）を diff 付きで提示する。既存カードと意味が同じなら CardId を維持して文面だけ更新し、学習履歴を保全する。

**3. 承認と反映（実装済みの承認フローを流用）**
提案への承認を得てから addNote / updateNoteFields / stale 化を実行し、成功分だけ state.json を更新する。部分承認（ページ単位・カード単位）を許容する。

**検知のトリガー**: 同期はstate突合によるpull型なので、`/anki` の実行が起点になる。加えて `/ingest`・`/query` がinsightページを作成・更新した際、作業サマリで「/anki での同期」を提案する1行を各スキルに追記する（自動実行はしない。提案のみ）。

- [x] **1. Ankiへの投入経路の技術調査**（最優先・要ユーザー確認と連動）
  - 候補: (a) AnkiConnect（デスクトップAnkiを起動した状態でHTTP API経由・ノート更新可） (b) genanki等で `.apkg` を生成しユーザーが手動インポート (c) AnkiWeb直接（公式APIなし・スクレイピングは不可）
  - 環境は WSL2。ユーザーのAnkiがどこで動いているか（Windows側デスクトップ/スマホのみ等）で実現性が変わる。WSL2からWindows側のAnkiConnect（localhost:8765）へ到達できるかの確認方法も調べる
  - 更新（同一ノートの上書き）・重複排除がそれぞれの経路でどう実現できるかを必ず確認する
  - **調査結果:**
    - **AnkiConnect**: Anki起動中に既定 `127.0.0.1:8765` で動作する。`addNote(s)`、`findNotes`、`notesInfo`、`updateNoteFields`、タグ更新を利用できる。追加時の標準的な重複判定は先頭フィールド中心なので、それだけに頼らず専用 `CardId` フィールドを検索して add/update を分岐する。即時反映・差分更新・note ID保存に最も向くが、デスクトップAnkiとアドオンが必要
    - **`.apkg` / genanki**: デスクトップAnkiへ手動インポートでき、常時API接続は不要。同一GUIDなら既存ノート更新に使えるが、モデルID・フィールド構成を固定する必要があり、インポート操作と結果確認は手動。削除同期にも向かない。初回配布・接続不能時の代替としては有効
    - **AnkiWeb直接**: 公開された公式APIがないため対象外。スクレイピングもしない
    - **WSL2疎通**: mirrored networkingならWSLから `http://127.0.0.1:8765` を試す。NATならWindowsホストIP（`ip route show` のdefault gateway）を使う必要があるが、AnkiConnectの既定bindはloopbackのみなので、そのままでは届かない。bindを外部向けに変える場合はLAN公開を避け、Windows FirewallでWSL側に限定し、可能ならAPI keyも設定する。まず `version` actionをPOSTして確認する
    - **実機確認（2026-07-19）**: WSLから `127.0.0.1:8765` への直接接続は不可。一方、WSLから `powershell.exe` を呼び、Windows側の `127.0.0.1:8765` へ問い合わせる経路ではAPI version 6が正常応答した。AnkiConnectを外部bindせずに済むため、このPowerShell中継を採用する。既存デッキは1件のみで `insight` はなく、既存ノートタイプにも `Insight Basic` / `Insight Cloze` はないことを確認した
    - **推奨**: WindowsデスクトップAnkiを使えるならAnkiConnect。使えない、またはアドオンを入れたくない場合は `.apkg`
    - **根拠**: [AnkiConnect API](https://github.com/FooSoft/anki-connect)、[Ankiテキスト取込の重複・GUID仕様](https://docs.ankiweb.net/importing/text-files.html)、[WSLネットワーク](https://learn.microsoft.com/en-us/windows/wsl/networking)、[genanki](https://github.com/kerrickstaley/genanki)
- [x] **2. カード設計のベストプラクティス調査**
  - SuperMemoの「20 rules of formulating knowledge」等、概念理解向けのカード設計原則を調べ、「品質基準（暫定）」を裏付け・修正する
  - Basic（表裏）とCloze（穴埋め）の使い分け基準。概念習得目的でClozeが有効な場面はどこか
  - **調査結果:** SuperMemoの中心原則は「理解してから記憶する」「最小情報」「穴埋め」「干渉回避」「表現の最適化」。本計画の1カード1事実は妥当だが、より正確には「1カード1想起・1採点点」とする。状況→概念やWhy/HowはBasic向き。Clozeは、十分な文脈を残して単一の関係（例: 原因→結果）や短い構成要素を思い出す場合には効率的だが、状況判断・概念選択・説明生成を問う主形式にはしない
  - **根拠**: [SuperMemo: Twenty rules of formulating knowledge](https://www.supermemo.com/en/blog/twenty-rules-of-formulating-knowledge)、[Anki Manual: Cloze Deletion](https://docs.ankiweb.net/editing.html#cloze-deletion)
- [x] **3. ノートタイプ・デッキ構成の設計**
  - カスタムノートタイプ（表・裏・出典スラグ・タグ）を作るか、標準Basicで済ませるか
  - デッキは1つ（`insight`）か、index.md のカテゴリ（説得・影響力/学習/習慣…）でサブデッキを切るか。タグで代替できるならデッキは1つがシンプル
  - **調査結果:** 標準Basicには安定ID・出典・非採点補足を安全に分離する場所がないため、専用ノートタイプを採用する。デッキはスケジュールと学習設定の単位、タグは分類・検索の単位と整理する。insightのカテゴリは学習設定を分ける必然性がないため、単一デッキ＋階層タグを既定にする。これによりカテゴリ再編でもノート移動を避けられる
  - **根拠**: [Anki Manual: Deck Options](https://docs.ankiweb.net/deck-options)、[Anki Manual: Tags](https://docs.ankiweb.net/editing.html#using-tags)
- [x] **4. 生成パイプラインの設計**
  - LLM（スキル実行時のエージェント）がページを読んでカード文面を書く前提で、生成の一貫性をどう担保するか（カード生成基準を references/ に切り出してスキルから参照する等、既存スキルの `essence-review-criteria.md` 方式に倣う）
  - 生成したカードの品質チェック（自己審査 or サブエージェント審査）を入れるか
  - **調査結果:** 生成基準は `anki/card-criteria.md` に独立させ、整数versionを付ける。処理を (1)ページ解析・型分類 (2)候補生成 (3)基準ごとの自己審査 (4)重複・干渉のページ横断検査 (5)ユーザー承認 (6)投入 に分離する。審査では各カードに「問いの一意性・1概念/1判断・根拠箇所・概念の過剰分割がないか・類似カード・Cloze適格性」を記録し、不合格は修正または破棄する。通常は同一エージェントの別パスによる自己審査で十分。全件初回生成や基準変更時だけ独立サブエージェント審査を任意で使う（通常実行に必須化しない）
- [x] **5. 既存の類似ツールの調査（任意）**
  - obsidian-to-anki 等、Markdown wiki → Anki の既存ツールで流用できる設計（ID埋め込み方式など）がないか
  - **調査結果:** Obsidian_to_AnkiはAnkiConnectが返したnote IDをMarkdown内のHTMLコメント等に埋め、再同期に使う。本wikiではソースページを生成物で汚さないため、ID埋め込み自体は採用せず `CardId` フィールド＋`state.json` の二重管理へ翻案する。同ツールはMarkdown記法からカードを抽出する設計であり、本件の「ページを意味解釈してカード生成」とは主目的が異なるため、直接依存はしない
  - **根拠**: [Obsidian_to_Anki](https://github.com/ObsidianToAnki/Obsidian_to_Anki)
